Generic Object Pipeline — Deep Walkthrough & Generalization Proposal

Every load, one path — traced through the real code with exact parameters, a worked P2P example showing what lands in each table, the discrepancies that stop it being one path today, and the concrete change that makes it one.

Scope. This document is a companion to the accepted Canonical per-object processing recipe (Coding Standards section 7.1 of docs/DMT_DESIGN.html). It goes one level deeper: it names the exact procedure signatures and parameters at every call site, shows the code inside each step with mock data, catalogs where objects diverge from the single pattern, and proposes the specific registry columns and code collapses that would make all objects follow one path. Nothing here has been changed in code — it is analysis and proposal for review.

Reference object. Suppliers is named by 7.1 as the proven template (live run 270, prefix 1143), so it defines the structural pattern. The worked example uses PurchaseOrders for the step-by-step data, because it has real one-to-many shape (one header → several lines) — a single-table object is too simple to show what "lands in each table." Every file-and-line reference is a clickable link that opens the source in VS Code at that line.

Step 3.3b — Transform-time error capture proposed 2026-07-16

The CSV→STG bulk loader and Step 3.2 (validate) both isolate bad rows without aborting the batch; the transform insert must do the same, and today it does not. The transform is a single set-based INSERT…SELECT. If one row can't be built — a garbage date, a value too long, a NOT-NULL miss — the statement raises, WHEN OTHERS re-raises, and the whole object fails, clean rows included. That is the last unguarded lane in the pipeline.

The three-part fix, symmetric with the load lane's LOG ERRORS standard (§7):

  1. Non-throwing SELECT. Every typed conversion carries DEFAULT NULL ON CONVERSION ERROR, so a bad value becomes NULL instead of raising — the error then surfaces at the column boundary where it can be caught.
  2. LOG ERRORS on the insert into one shared errlog, DMT_TFM_ERRLOG (5 control columns + STG_SEQUENCE_ID — the only data column, common to every TFM table, so one table serves all objects). Bad rows divert; good rows land.
  3. Fold + flag. This run+object's diverted rows are folded into DMT_STG_TFM_ERROR_TBL as [TRANSFORM_ERROR] with the real ORA- text, then FLAG_STG_FAILED(p_run_id) flips their STG rows FAILED — the same helper the validator uses, keyed on "has an error row this run," not on the tag.

The residual case — a statement-level throw LOG ERRORS can't trap (an error raised inside the SELECT projection despite rule 1, e.g. a scalar sub-query returning two rows). The transform's WHEN OTHERS is the fallback: re-process this object's rows one at a time so the true culprit is isolated and the clean rows still load — and if even that can't proceed, mark the whole candidate batch [TRANSFORM_ERROR] with SQLERRM so nothing is left unaccounted. Fast path for the 99%, slow row-by-row only for the poisoned batch.

Step 3.5 — Reconcile + the accounting gate Rule #1

Move 2 — Standardize the four package signatures (Part B, right column)

Uniform GENERATE_FBDI (5-arg) and uniform RECONCILE_BATCH (always carry p_cemli_code) are the two that unblock generic dispatch. Multi-record objects keep their internal sub-steps (PO headers/lines/locs/dists) inside their one registered TRANSFORM/GENERATE_FBDI, so the engine still sees one call.
