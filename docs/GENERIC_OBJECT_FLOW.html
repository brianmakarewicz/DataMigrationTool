Generic Object Pipeline — Standard Flow & Per-Object Contract

One path for every object. This document does two jobs: it teaches how a load travels end to end (with worked examples and real table snapshots), and it states the contract — the exact registry rows, tables, packages, procedures, and parameters that every object must provide.

Who this is for, and how to read it.
  • To understand the pipeline (a developer or an LLM learning the system): read Part A. It follows one real run, step by step, and shows what each table holds before and after each step.
  • To build or wire up a new object (a developer or an LLM writing code to spec): read Part B and follow Part C. Part B is the contract — every artifact an object must expose, with each procedure's full signature written one parameter per line. Part C is the same contract as an ordered checklist.
This document describes the standard flow — the single path an object is expected to follow. Where existing code has not yet caught up to the standard, that drift is listed once, in the Appendix, and left for a later revision. Everything else here is normative.

Part A · The standard flow, step by step

A load travels one path: caller → submission → run & queue rows → heartbeat → worker → shared engine → Fusion → reconcile → accounting gate. The shared engine is one function, DMT_LOADER_PKG.run_one_object_type (db/packages/dmt_loader_pkg.pkb.sql:993). Every object's runner is three lines that call it. Read this part to understand the machine; Part B turns each step into a build requirement.

Entry — the caller chooses what to run

Every run begins with a person choosing which objects to migrate. That choice is a single string of pipeline codes, passed as p_pipeline_codes. A pipeline code is either the name of a pipeline that expands to several objects, or one object written as STANDALONE:ObjectName. The same entry point therefore covers every scope the user might want:

What the user wants to runp_pipeline_codesObjects that get queued
Just suppliersSTANDALONE:SuppliersSuppliers only
Just purchase ordersSTANDALONE:PurchaseOrdersPurchaseOrders only
Just customersSTANDALONE:CustomersCustomers only
The entire P2P flowP2PEvery object registered to the P2P pipeline, in SORT_ORDER — Suppliers first, then PurchaseOrders, and so on

Everything the engine needs is one static table, DMT_PIPELINE_DEF_TBL — one row per object. It records which pipeline the object belongs to, its order within that pipeline, what it depends on, and which procedures load and reconcile it. A few of its seeded rows (db/seed/dmt_pipeline_def_tbl.sql):

A named code like P2P is expanded into its object list by the function GET_CEMLI_SEQUENCE in package DMT_SCHEDULER_PKG (dmt_scheduler_pkg.pkb.sql:16). It reads the same registry — no object names appear in the code:

FUNCTION GET_CEMLI_SEQUENCE (
    p_pipeline_code IN VARCHAR2
) RETURN VARCHAR2 IS
    l_seq VARCHAR2(4000);
BEGIN
    SELECT LISTAGG(CEMLI_CODE, ',') WITHIN GROUP (ORDER BY SORT_ORDER)
    INTO   l_seq
    FROM   DMT_OWNER.DMT_PIPELINE_DEF_TBL
    WHERE  PIPELINE_CODE = UPPER(p_pipeline_code)   -- 'P2P'
    AND    EXEC_PROC IS NOT NULL;
    RETURN l_seq;   -- 'Items,Suppliers,SupplierAddresses,...,PurchaseOrders,...'
END GET_CEMLI_SEQUENCE;

That comma-separated list is what the submission procedure walks next, inserting one DMT_WORK_QUEUE_TBL row per object. A STANDALONE:Suppliers code skips this lookup — the object name is taken directly.

What submission writes — two tables, and only these two:

  • One row in DMT_PIPELINE_RUN_TBL — the run itself: a new RUN_ID, the scenario, the run mode, the prefix, and the on-failure policy.
  • One row per object in DMT_WORK_QUEUE_TBL — the work list. Each row carries its CEMLI_CODE, its SORT_ORDER, and its DEPENDS_ON. A row with no dependency starts READY; a row that depends on another object starts PENDING and waits (dmt_scheduler_pkg.pkb.sql:255).

After the commit the run exists on paper and nothing has run. From here on, the heartbeat moves it forward.

The heartbeat tick — the clock that drives every run

Procedure: DMT_QUEUE_PKG.HEARTBEAT_TICK — dmt_queue_pkg.pkb.sql:534. It is a background database job that fires every 60 seconds. It is deliberately lightweight: it checks statuses and spawns short child jobs, and it makes no Fusion calls itself. The slow work — talking to Fusion, generating files — always runs inside the child jobs the tick spawns, never on the tick.

Each tick runs the same seven phases in order. Every run in flight advances by whatever these phases can do in one pass:

#PhaseWhat it does
1run_preflightsRuns a new run's preflight checks (kept off the tick itself by spawning the live Fusion calls).
2promote_readyFlips a PENDING row to READY once every object named in its DEPENDS_ON is done. This is where PurchaseOrders becomes runnable after Suppliers finishes.
3dispatch_ess_pollsChecks on Fusion load jobs that are already running and advances the rows waiting on them.
4dispatch_readyPicks up READY rows and starts them — the dispatch step detailed below.
5dispatch_reconcileStarts reconciliation for rows whose load has finished.
6handle_failuresApplies the on-failure policy and cascades a SKIP to the dependents of a failed object.
7rollup_run_statusesRecomputes each run's overall status from the state of its work rows.

Nothing pushes a run along except this tick. Submission leaves rows sitting in the queue; the next tick — within 60 seconds — is what actually starts the work.

1Dispatch — pick the object and start its procedure

Dispatch happens in two procedures. First dispatch_ready (phase 4 of the tick) chooses the next object and hands it to a background job. Then that job runs EXECUTE_ONE, which looks up the object's procedure and calls it.

The one thing to hold onto while reading: dispatch_ready never learns the object's procedure name. It only knows the queue id. It carries that id across to a fresh background session, and the name lookup happens over there, inside EXECUTE_ONE. Follow the queue id (5002) through the four blocks below.

Step 1a · Choose and claim the object — dispatch_ready

dmt_queue_pkg.pkb.sql:258. This is a cursor FOR loop: the SELECT in the parentheses is the cursor, and rec is the row variable the loop fills on each pass — one READY queue row at a time, lowest SORT_ORDER first.

FOR rec IN (
    SELECT q.QUEUE_ID, q.RUN_ID, q.CEMLI_CODE, q.PARTITION_KEY
    FROM   DMT_WORK_QUEUE_TBL q
    JOIN   DMT_PIPELINE_RUN_TBL r ON r.RUN_ID = q.RUN_ID
    WHERE  q.WORK_STATUS = 'READY'
    AND    r.PREFLIGHT_STATUS = 'OK'
    ORDER BY q.SORT_ORDER            -- lowest order first
) LOOP
    -- rec is one READY row: rec.QUEUE_ID = 5002, rec.CEMLI_CODE = 'PurchaseOrders', ...

    UPDATE DMT_WORK_QUEUE_TBL           -- claim it so the next tick skips it
    SET    WORK_STATUS = 'PROCESSING', STARTED_AT = SYSTIMESTAMP
    WHERE  QUEUE_ID = rec.QUEUE_ID;
    COMMIT;

    spawn_child('DMT_WQ_' || rec.QUEUE_ID,                                -- 'DMT_WQ_5002'
        'BEGIN DMT_OWNER.DMT_QUEUE_WORKER_PKG.EXECUTE_ONE(' || rec.QUEUE_ID || '); END;');
END LOOP;

The row is claimed — flipped to PROCESSING and committed so the next 60-second tick will not re-pick it — and then handed to spawn_child. Notice what is passed: the job name DMT_WQ_5002 and a string of PL/SQL, BEGIN ... EXECUTE_ONE(5002); END;. Only the queue id 5002 is baked into that text. No procedure name, no run details — just the id.

Step 1b · Hand off to a background job — spawn_child

dmt_queue_pkg.pkb.sql:71. It takes the two strings it was given and creates a one-shot Oracle Scheduler job that runs that PL/SQL text in its own background session:

PROCEDURE spawn_child (
    p_job_name IN VARCHAR2,
    p_plsql    IN VARCHAR2
) IS
BEGIN
    DBMS_SCHEDULER.CREATE_JOB(
        job_name   => p_job_name,     -- 'DMT_WQ_5002'
        job_type   => 'PLSQL_BLOCK',
        job_action => p_plsql,        -- 'BEGIN ... EXECUTE_ONE(5002); END;'
        enabled    => TRUE,           -- starts running at once
        auto_drop  => TRUE);          -- deletes itself when finished
END spawn_child;

That is the whole handoff. The heartbeat tick is now free — it did not run the load, it only launched a job that will. From here the work runs in that separate session, starting with EXECUTE_ONE(5002). (This separation is also why the worker lives in its own package: a child job that depended back on the heartbeat's package would risk a library-cache deadlock.)

Step 1c · Look up the object's procedure — EXECUTE_ONE → get_dispatch

dmt_queue_worker_pkg.pkb.sql:293. Now in the background session, EXECUTE_ONE gets only the queue id. It reads the full queue row and run row into record variables, then calls get_dispatch to fill four OUT variables from the registry:

PROCEDURE EXECUTE_ONE (
    p_queue_id IN NUMBER
) IS
    l_rec         DMT_WORK_QUEUE_TBL%ROWTYPE;     -- the whole queue row
    l_run_rec     DMT_PIPELINE_RUN_TBL%ROWTYPE;
    l_exec_proc   VARCHAR2(200);                     -- get_dispatch fills these ↓
    l_exec_mode   VARCHAR2(10);
    l_recon_proc  VARCHAR2(200);
    l_recon_cemli VARCHAR2(1);
BEGIN
    SELECT * INTO l_rec     FROM DMT_WORK_QUEUE_TBL   WHERE QUEUE_ID = p_queue_id;
    SELECT * INTO l_run_rec FROM DMT_PIPELINE_RUN_TBL WHERE RUN_ID   = l_rec.RUN_ID;

    -- registry lookup: reads DMT_PIPELINE_DEF_TBL for this object's CEMLI_CODE
    get_dispatch(l_rec.CEMLI_CODE, l_exec_proc, l_exec_mode, l_recon_proc, l_recon_cemli);
    -- l_exec_proc is now 'DMT_LOADER_PKG.RUN_PURCHASE_ORDERS'

The queue id finally became a name here. l_rec.CEMLI_CODE is 'PurchaseOrders' (read from the queue row), and get_dispatch uses it as the key. That procedure is the registry SELECT keyed on CEMLI_CODE, writing its answers back through the OUT parameters (dmt_queue_worker_pkg.pkb.sql:15):

PROCEDURE get_dispatch (
    p_cemli_code          IN  VARCHAR2,
    x_exec_proc           OUT VARCHAR2,
    x_exec_mode           OUT VARCHAR2,
    x_recon_proc          OUT VARCHAR2,
    x_recon_has_cemli_arg OUT VARCHAR2
) IS
BEGIN
    SELECT EXEC_PROC, EXEC_MODE, RECON_PROC, RECON_HAS_CEMLI_ARG
    INTO   x_exec_proc, x_exec_mode, x_recon_proc, x_recon_has_cemli_arg
    FROM   DMT_PIPELINE_DEF_TBL
    WHERE  CEMLI_CODE = p_cemli_code;   -- 'PurchaseOrders'
END get_dispatch;

So l_exec_proc now holds the plain string 'DMT_LOADER_PKG.RUN_PURCHASE_ORDERS'. If the object had no registry row, get_dispatch returns NULL and EXECUTE_ONE raises a clear error (ORA-20100) instead of guessing.

Step 1d · Build and run the call — invoke_registered

Still in EXECUTE_ONE, the looked-up name is passed straight into invoke_registered — this is where p_proc comes from. It is l_exec_proc, the value get_dispatch just read from the table:

    DMT_LOADER_PKG.g_async_mode := (l_exec_mode = 'ASYNC');

    invoke_registered(
        p_proc          => l_exec_proc,          -- <-- the looked-up name becomes p_proc
        p_style         => 'EXEC',
        p_run_id        => l_rec.RUN_ID,
        p_cemli_code    => l_rec.CEMLI_CODE,
        p_scenario_name => l_run_rec.SCENARIO_NAME,
        p_run_mode      => l_run_rec.RUN_MODE);

Inside invoke_registered (dmt_queue_worker_pkg.pkb.sql:62), p_proc is that name. Because it is only known at run time — the point of the registry — the call is assembled as text and run with EXECUTE IMMEDIATE. The name is first checked against a strict PKG.PROC pattern so no unexpected text can reach the block, and every value is passed as a bind variable:

IF NOT REGEXP_LIKE(p_proc, '^[A-Z][A-Z0-9_$#]*\.[A-Z][A-Z0-9_$#]*$') THEN
    RAISE_APPLICATION_ERROR(-20102, '... not a valid PKG.PROC identifier');
END IF;

EXECUTE IMMEDIATE
    'BEGIN ' || p_proc || '(p_run_id => :b1, p_scenario_name => :b2, '
    || 'p_run_mode => :b3, p_skip_bu_refresh => TRUE); END;'
    USING p_run_id, p_scenario_name, p_run_mode;

So for PurchaseOrders the statement that actually runs is BEGIN DMT_LOADER_PKG.RUN_PURCHASE_ORDERS(p_run_id => :b1, ...); END;. Trace the whole chain: rec.QUEUE_ID (5002) → baked into the job text → EXECUTE_ONE(5002) → l_rec.CEMLI_CODE ('PurchaseOrders') → get_dispatch → l_exec_proc → p_proc → the built EXECUTE IMMEDIATE. The object name came from the table at every step, never from the worker code. There is no per-object branch anywhere in this path.

After the call returns: if the loader left a Fusion job id in g_load_ess_id, the row moves to AWAITING_LOAD and later ticks poll it (dmt_queue_worker_pkg.pkb.sql:385); otherwise it routes to reconciliation or straight to the accounting gate.

Step 3.5 — Reconcile + the accounting gate Rule #1

When the ESS jobs finish, the heartbeat calls DMT_QUEUE_WORKER_PKG.RECONCILE_ONE(p_queue_id) (dmt_queue_worker_pkg.pkb.sql:438), which invokes the registered reconciler through invoke_registered, then calls the single terminal apply_accounting_gate(p_queue_id, p_run_id, p_cemli_code) (dmt_queue_worker_pkg.pkb.sql:241) — the only writer of WORK_STATUS='DONE'. The gate's counts come from ACCOUNT_ROWS (dmt_queue_worker_pkg.pkb.sql:134), which builds its SQL from the catalog rows for the object.

Part B · The per-object contract

This part is normative. To add an object to the pipeline you must provide every artifact below. Nothing here requires editing the engine, the worker, or the scheduler — the object is defined entirely by registry rows, tables, and four packages plus a runner whose signatures match exactly. Each required signature is written one parameter per line; match the parameter names and defaults.

B.3 — The four packages + the runner

Every object ships four packages — a validator, a transform, a generator, and a results (reconciler) — plus a three-line runner in DMT_LOADER_PKG. The generator is _FBDI_GEN_PKG for FBDI objects (_HDL_GEN_PKG / _FBL_GEN_PKG for HCM / configuration). Each must expose the entry point below with exactly this signature.

Required · the runner — registered as EXEC_PROC. Three lines: resolve the scenario, call the engine with the object's CEMLI code.
PROCEDURE RUN_<OBJECT> (
    p_run_id          IN NUMBER,
    p_scenario_name   IN VARCHAR2 DEFAULT NULL,
    p_run_mode        IN VARCHAR2 DEFAULT 'NEW',
    p_skip_bu_refresh IN BOOLEAN  DEFAULT FALSE
);
Required · the validator — DMT_<STEM>_VALIDATOR_PKG. One entry point that flips bad STG rows to FAILED with a [PRE_VALIDATION] tag before transform.
PROCEDURE VALIDATE_PRE_TRANSFORM (
    p_run_id          IN NUMBER,
    p_scenario_id     IN NUMBER   DEFAULT NULL,
    p_run_mode        IN VARCHAR2 DEFAULT 'NEW',
    p_validate_upstream IN VARCHAR2 DEFAULT 'N'
);
-- Set p_validate_upstream ON when the object depends on same-run data

-- exist in Fusion. Objects that filter by document/type may add a single
-- p_doc_type_filter / p_inv_type_filter parameter.
Required · the transform — DMT_<STEM>_TRANSFORM_PKG. One entry point that copies STG → TFM as STAGED, applies the prefix, links children to headers, and flips STG rows to TRANSFORMED. Multi-record objects keep their sub-steps (headers/lines/locations/distributions) inside this one call.
PROCEDURE TRANSFORM (

PROCEDURE RECONCILE_BATCH (

Why no engine edit is needed. The worker's dispatch (get_dispatch + invoke_registered), the single accounting gate, and the catalog-driven ACCOUNT_ROWS / MARK_GENERATED_ROWS_FAILED read everything about an object from the rows and packages above. Status literals (STAGED/GENERATED/LOADED/FAILED) and the p_run_id parameter name are uniform across all four packages. Provide the rows, tables, and matching signatures and the object flows the Part A path with no code change to the shared machinery.

Part C · New-object checklist

The contract of Part B as an ordered build list. Each item links to its Part B requirement.

  1. Add one row to DMT_PIPELINE_DEF_TBL (B.1) — PIPELINE_CODE, CEMLI_CODE, SORT_ORDER, DEPENDS_ON, EXEC_PROC, EXEC_MODE, RECON_PROC, RECON_HAS_CEMLI_ARG, POSTRUN_JOB.
  2. Add one row per TFM record type to DMT_CEMLI_CATALOG_TBL (B.1) — CEMLI_CODE, DISPLAY_NAME, TFM_TABLE, STATUS_COLUMN, ROW_FILTER, SORT_ORDER.
  3. Seed DMT_ERP_INTERFACE_OPTIONS_TBL (job name, UCM account, interface-details id, optional credential override) and DMT_BIP_REPORT_TBL (reconciliation report path).
  4. Create DMT_<STEM>_STG_TBL / _TFM_TBL with the standard status columns (STG_STATUS / TFM_STATUS) and ERROR_TEXT (B.2).
  5. Build the four packages to the exact signatures in B.3 — one validator entry, one transform entry (sub-steps internal), one generator, one RECONCILE_BATCH.
  6. Add the three-line RUN_<OBJECT> to DMT_LOADER_PKG and register it as EXEC_PROC.
  7. Decide p_validate_upstream's default (ON if the object depends on same-run data — like PO on Suppliers; OFF if its references already exist in Fusion).
  8. Deploy the BIP data model + report to /Custom/DMT2/; write objects/<Name>/README.md.
  9. Run end-to-end under a test prefix — GOOD rows reach LOADED, BAD rows reach FAILED with reportable ERROR_TEXT (Rule #1). No engine edit at any step.

Appendix · Known drift from this standard revision deferred

The standard above is the target every object should follow. Some existing code has not fully caught up — mostly hardcoded per-object branches inside the engine that predate the catalog-driven dispatch, and a few package signatures that vary from B.3. These gaps are recorded here so nothing is lost, but they are out of scope for this document's purpose and left for a later revision. Severity: high blocks generalization / risks correctness · med drift & maintenance cost · low cosmetic.
