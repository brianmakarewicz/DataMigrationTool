Overview & status legend VERIFIED 2026-07-04
Single source of design truth for the Data Migration Tool — DB schema, pipeline, error handling, packages, and the APEX UI that sits on top. Review here before building so code does not diverge from intent.
Purpose. The point of the tool is not to pre-validate all the data (although some validation is possible) — it is to move all available data into the cloud and ensure that every record is accounted for through a positive success or a positive failure message. The tool gives users one place to put data, and hides the complexity of running the loads, re-running the same data, and reporting on different loads that could otherwise load the same data twice.
DMT moves data from source systems (Oracle EBS or generic CSV) into Oracle Fusion Cloud through a multi-stage pipeline. Two load patterns: FBDI (CSV, ERP modules) and HDL (DAT, HCM). Everything is pure PL/SQL on ATP (schema DMT_OWNER); APEX is the UI layer.
Ways to get data in.
CSV upload with proprietary load syntax — users send in data shaped like the FBDI, but column order is determined by the file header rather than fixed position. IMPLEMENTED
FBDI structured upload — users upload files in the exact FBDI-generated format: position-based columns, no header row, as produced by the Oracle FBDI templates. Both this and the proprietary format are accepted by the same upload path. IMPLEMENTED
Direct DB load — insert straight into the STG tables (EBS DB-link push or SQL). The EBS push comes from the separate EBSAdaptors repo through a three-object interface contract — see §4. IMPLEMENTED
REST APIs exposing the STG tables for external loaders. TO-DO (future state)
AI-assisted loading — allow an AI agent to move data into the STG tables, or expose an MCP server over the STG layer. TO-DO (next stage after REST)
Pipeline stages defined
Worked example: follow two real records through every stage and status change →
Stage | What it does | Tables touched |
Stage | Raw source rows land in staging — via CSV upload (header-driven syntax), EBS DB-link push, or direct SQL. Rows arrive STATUS='NEW', tagged to their mandatory scenario. No prefix, no run association yet. | DMT_{obj}_STG_TBL |
Pre-validate | Runs on STG rows before anything else. Two distinct checks:
(1) Cross-object dependency — has the master data this record needs (a different object, loaded in an earlier run) gone all the way into Fusion? Requires the upstream record to have a LOADED TFM row from any run (decided 2026-07-07) — within one run, the dependency graph guarantees the parent work item finished first, so just-loaded parents qualify; in a failed-items re-run, parents loaded in earlier runs qualify. The match compares the source values as they appear in the data (no prefix) on both staging sides, then joins to the parent's TFM outcome. The reference written into the load file carries the prefix of the parent's most recent LOADED row — automatically correct for same-run, prior-run, and mixed cases; the submission screen's dependent-run picker is an optional override to pin a specific run. This validation can be turned off per run ("Validate upstream dependencies", default ON, section 8) — with it off, rows proceed and a genuinely missing parent still fails at Fusion with a reportable error. Example: a PO header requires its supplier LOADED. Enforces "no transactional record without loaded master data".
(2) Within-object cascade — header, lines, locations, and distributions travel together in the same FBDI, so there is no Fusion check between them. Instead, if a parent row fails pre-validation in this pass, the failure cascades down (header → lines → locations → distributions) so no orphan children enter the file.
Failing rows go FAILED [PRE_VALIDATION] and never enter the run. | STG (status/error updates only) |
Transform | Copies each surviving STG row into the object's TFM table: applies the run prefix to business keys, derives/defaults fields, maps source values to Fusion values. The TFM row is stamped with RUN_ID and STG_SEQUENCE_ID (lineage back to source). Failures are recorded in the stage→transform result table with [TRANSFORM_ERROR] (section 5) — the staging row's status is set to FAILED (the message never goes on the staging row — it lives in the error table). Success is denoted by the TFM row itself — run-stamped and timestamped; this error table records failures only (section 5). | STG → DMT_{obj}_TFM_TBL |
Post-validate | Runs on TFM rows after transform. Checks the content about to be sent to Fusion: required fields present, values in allowed sets, cross-field rules. Failing rows go FAILED [POST_VALIDATION] on their TFM row and are excluded from the file — nothing is written back to staging. Coverage is intentionally light — Fusion is the final judge; the pipeline's job is to account for every outcome. | TFM (status/error updates) |
Generate FBDI/HDL | Writes surviving TFM rows into FBDI CSV(s) (position-based, no header) or an HDL DAT file, then zips. Grouped objects split into one file set per partition (BU/ledger/org). TFM rows become GENERATED with FBDI_CSV_ID pointing at the exact file they went into. | TFM → DMT_FBDI_CSV_TBL → DMT_FBDI_ZIP_TBL |
Load (ESS) | Uploads the ZIP to Fusion UCM and submits the ESS load/import job(s) with the object's ParameterList. ESS request IDs are recorded on the work item. | DMT_WORK_QUEUE_TBL, DMT_ESS_JOB_TBL |
Poll | The heartbeat checks ESS job state each tick until the job hierarchy reaches a terminal state (SUCCEEDED / WARNING / ERROR). WARNING from SqlLdr means zero rows committed — treated as a failure signal, not success. | DMT_ESS_JOB_TBL, DMT_WORK_QUEUE_TBL |
Reconcile (BIP) | Runs a BIP report against the Fusion interface and base tables, joins results back to TFM rows by business key (prefix + key within the RUN_ID). A row found in base tables → LOADED + its FUSION_*_ID. A row rejected in the interface → FAILED [FUSION_ERROR] with the interface error text. Nothing is written back to staging — the TFM row is the sole record of the Fusion outcome (decided 2026-07-07). 0 BIP rows is never assumed to mean success. | Fusion (read) → TFM only |
LOADED / FAILED | Terminal row states. Every record that entered the run must end in exactly one: positive success (in Fusion base tables, ID captured) or positive failure (a reportable error attributed to that row). Anything else is unaccounted for and fails the work item. | — |
Three independent status levels — never conflate them
Status is tracked at three altitudes, each with its own column and vocabulary. A run is one submission of the pipeline (one or more objects); a work item is one unit of queued work inside a run (typically one object, or one FBDI partition of an object); a row is one source record moving through the STG and TFM tables. A run can be COMPLETED while individual rows are FAILED; a row is only ever judged by its own STATUS.
Run — RUN_STATUS · DMT_PIPELINE_RUN_TBL
Status | Meaning |
QUEUED | Run submitted; waiting for the queue poller to pick it up. |
IN_PROGRESS | At least one work item is not yet finished (waiting, processing, or polling). |
COMPLETED | All work items finished; every record accounted for with no failures. |
COMPLETED_ERRORS | All work items finished; some rows ended FAILED (with reportable errors). |
FAILED | The run itself could not finish — an unaccounted-for record or infrastructure failure. Work items SKIPPED because of that failure don't change the status further: their records were never selected, so they are untouched, not unaccounted (decided 2026-07-07). |
NO_ROWS_PROCESSED | Run finished and every work item selected zero rows — nothing in the whole run matched the scenario/mode. An individual zero-row item is simply DONE with zero counts (grey tile); it does not make the run NO_ROWS_PROCESSED (decided 2026-07-07). |
Work item — WORK_STATUS · DMT_WORK_QUEUE_TBL
Status | Meaning |
PENDING | Created; waiting for its DEPENDS_ON predecessor to finish. |
READY | Dependencies satisfied; eligible for pickup by the heartbeat. |
PROCESSING (end state — code/UI still say VALIDATING, P2 §12) | A child job is running the item's data phase end-to-end: pre-validate the object's STG rows (dependency checks) → transform STG → TFM (prefix, derivations) → post-validate TFM rows (content checks) → generate the FBDI/HDL file (TFM → DMT_FBDI_CSV_TBL → DMT_FBDI_ZIP_TBL) → submit the load to Fusion. One status covers the whole phase because it runs as a single child job (DMT_QUEUE_WORKER_PKG.EXECUTE_ONEsrc); the item leaves PROCESSING when the load is submitted (→ LOADING/AWAITING_LOAD) or the phase fails. See "Pipeline stages defined" above for what each step checks. |
LOADING | Load submission in progress (ZIP → UCM upload → ESS job submit). |
AWAITING_LOAD | ESS load job submitted; heartbeat is polling for completion. |
AWAITING_IMPORT | Import ESS job submitted; polling. |
AWAITING_POSTRUN | Post-run ESS step (per-object post-processing job) submitted; polling. Today only Assets has one (per-book Prepare + Post); the per-object post-run job is a catalog column (section 6). Distinction, stated to remove ambiguity: a post-run job is an ESS job that further processes the imported data. Reporting and reconciliation steps — including Projects' post-import reporting and every BIP validation report — are not post-run jobs; they belong to the Reconcile phase. |
RECONCILING | BIP reconciliation child job is running. |
DONE | Item finished and every row is accounted for — each row ended either LOADED (confirmed in Fusion base tables) or FAILED with a reportable error. Rows may have failed; DONE means nothing is unexplained. |
FAILED | Item ended with at least one row unaccounted for (no positive success and no positive failure — e.g. reconciliation could not resolve it), or the item hit an unrecoverable infrastructure error. An ESS job ERROR does not short-circuit to FAILED — the item still reconciles, and only goes FAILED if rows remain unexplained. |
SKIPPED | Not run because a dependency failed. |
The accounting rule. A work item is DONE if and only if every record is accounted for (base-table LOADED or interface-errored FAILED); it is FAILED only if any record is unaccounted for. Row failures alone never fail the work item — unexplained rows do.
Row — STATUS · STG & TFM tables
Layer | Status | Meaning |
STG | NEW | Loaded into staging; no run has attempted it yet. |
STG | RETRY RETIRED | Retired (2026-07-07): run modes drive selection predicates directly, so no re-queue status is needed — the staging status is forward-only (NEW → TRANSFORMED or FAILED) and is never reset. Exists in current code only; removed by the mode-predicate change (P3, section 12). |
STG | TRANSFORMED | The row's most recent stage→transform attempt succeeded — it was copied into TFM. Terminal from staging's point of view: whatever happens downstream (loaded or rejected in Fusion) is recorded on the TFM row and is never written back here. |
STG | FAILED | The row's most recent stage→transform attempt failed (pre-validation or transform). The why is in the stage→transform error table — never on this row. Only the stage→transform step writes this column, and a later successful retry flips it to TRANSFORMED (decided 2026-07-07). |
TFM | STAGED | Row created in TFM by the transform. |
TFM | GENERATED | Row written into an FBDI CSV / HDL DAT (FBDI_CSV_ID set). |
TFM | LOADED | BIP reconciliation confirmed the row in Fusion base tables (FUSION_*_ID captured). |
TFM | FAILED | Positive failure — validation, generation, load, or interface rejection; ERROR_TEXT attributed to this row. |
both | UNRECONCILED (derived — not stored) | Computed by the run-detail views (DMT_V_CEMLI_STATUSsrc, DMT_OBJECT_DETAIL_Vsrc, DMT_RECORD_DETAIL_Vsrc) when a row is FAILED with no ERROR_TEXT: the pipeline got neither a positive Fusion ID nor an error for it. This is the "unaccounted for" state — it counts as in-flight in the funnel rollup and fails the work item's accounting rule if it persists. End state (2026-07-07): no process may set FAILED without appending ERROR_TEXT in the same statement — the centralized [RECONCILE_ERROR] sweep guarantees this on the reconcile path — so this bucket exists purely as a defect detector and must trend to zero. (The legacy per-object DMT_*_IMP_RPT_V views computed the same bucket; they are invalid, unreferenced, and slated for deletion — §12.) |
How users run and re-run data — run modes + prefixes
Three run modes (RUN_MODE on the run), selected at submission. Combined with prefixes, they cover the real load lifecycle:
Mode | Selects | Typical use case |
NEW (default) | Staging rows with status NEW — never yet attempted. TRANSFORMED and FAILED rows are untouched. (RETRY and the ALL-mode reset are retired — run modes drive the selection predicates directly.) | Incremental loading. Load 10% of the data as a sanity check and run it; once it proves out, load the other 90% and run again — only the new rows are picked up. Repeat as more data arrives. |
FAILED | Two sources, one union: staging rows whose status is FAILED (died before the transform table), plus staging rows whose most recent TFM attempt ended FAILED (rejected downstream — staging still says TRANSFORMED because downstream outcomes are never written back to staging). Must use the most recent attempt: a record that failed in run 57 and loaded in run 58 is not selected. NEW rows are ignored. | Retry after a fix. 100% loaded but 20% failed on a Fusion setup miss; fix the setup, run FAILED mode — just the failures re-run, the 80% that loaded stay untouched. |
ALL (requires a scenario) | Every row in the scenario, selected directly by the run-mode parameter in the selection predicates (decided 2026-07-07) — no status reset, no extra procedure. The run still gets a new prefix. Current code instead calls reset_scenario_status to flip rows to a RETRY status — replace with mode-driven predicates and retire both the procedure and the RETRY status (P3, section 12). | Full regression re-run. After fixing the setup miss you want to reload all 100% to prove the fix didn't break the rows that previously worked. The new prefix keeps the re-load from colliding with the earlier load in Fusion. |
Why prefixes make this safe. Every test run stamps a fresh numeric prefix onto business keys at transform, so the same source records can be loaded into Fusion repeatedly — each run creates distinct Fusion records, and reconciliation joins each run's rows by its prefix, so runs never blur together. A production cutover runs once, without a prefix, loading the data as-is.

Glossary
Term | Meaning |
CEMLI | Oracle's term for a customization work item (Configuration/Extension/Modification/Localization/Interface). In this document a "CEMLI code" is simply an object's unique identifier, e.g. PurchaseOrders. |
FBDI | File-Based Data Import — Oracle Fusion's bulk-load mechanism for ERP data: position-based CSV files zipped and submitted to an import job. |
HDL | HCM Data Loader — Fusion's bulk-load mechanism for HCM (Human Capital Management) data: pipe-delimited .dat files submitted over REST. |
FBL | File-Based Loader for functional setup/configuration objects — pipe-delimited CSVs with headers (see open question on its load mechanism, section 13). |
ESS | Enterprise Scheduler Service — Fusion's job scheduler. Every load/import runs as an ESS job with a request id. |
BIP | Business Intelligence Publisher — Fusion's reporting engine. DMT runs BIP reports over Fusion tables to reconcile what actually loaded. |
UCM | Universal Content Management — Fusion's file store; FBDI ZIPs are uploaded there before the import job runs. |
ATP | Oracle Autonomous Transaction Processing — the cloud database DMT runs on (schema DMT_OWNER). |
APEX | Oracle Application Express — the low-code web framework the DMT user interface is built in. |
SqlLdr | SQL*Loader — the Fusion-side step that reads the CSV into interface tables during a load job. A SqlLdr WARNING means zero rows were committed. |
P2P / O2C / FINANCIALS / PROJECTS / HCM / CONFIGURATION | Pipeline codes (canonical values of PIPELINE_CODE, exactly as in the section 6 seed — screens may display friendlier names) — named sequences of objects run together: Procure-to-Pay, Order-to-Cash, Financials, Projects, Human Capital Management, Configuration. |
EPBCS | Enterprise Planning & Budgeting Cloud Service — the target for planning budgets. |
CTL | SQL*Loader control file — defines an FBDI CSV's exact column order. |
IR | Interactive Report — APEX's sortable/filterable grid component. |
MCCS | The predecessor conversion project (a completed client engagement) whose codebase informed the coding standards (section 7). |
P1 / P2 / P3 | Backlog priorities: P1 = fix ASAP (blocks correctness or rebuild), P2 = important, schedule soon, P3 = worthwhile cleanup. |
Sections by | Column header in the object tables: the object's partitioning dimension — one work item + one generated file set per value ("single" = no split; "Procurement BU" = one per business unit; "Ledger" = one per ledger). The partition columns themselves are configured in DMT_CEMLI_SPLIT_CFG. |
Scenario | A named dataset tag on every staging row (mandatory at ingestion). Used to scope runs to one dataset; required for ALL-mode re-runs. Production data is tagged too: a cutover run uses its own scenario, conventionally named PRODUCTION — a scenario like any other, so ALL-mode re-runs work for cutover exactly as for tests (decided 2026-07-07). |
f155.sql / app N | An APEX application export file — "f155.sql" is the full export of application 155; citations like "f155:60626" are line references into that export, used as evidence in the Appendix. Historical only — the highest-numbered app is always current. |
Rapid Implementation | Oracle's setup-spreadsheet bulk upload for configuration data (Setup and Maintenance → "Create … in Spreadsheet" tasks). Rejected for Banks partly for its documented weakness adding branches to existing banks (Oracle support notes 2234593.1 / 1923921.1). |
STG / TFM | Staging / Transform — the two table layers every record passes through: STG holds the raw source rows; TFM holds the run-specific transformed copies that become the load file. |
ParameterList | The ordered argument string an ESS import job requires at submission (business unit, ledger, options — different per job). Stored per object with the ERP interface options; a wrong ParameterList is the classic cause of a job stuck in WAIT. |
Interface tier / Base tier | The two places a record can be seen in Fusion during reconciliation: the interface tables (arrived, possibly rejected — "Tier 1") and the base tables (fully imported — "Tier 2"). LOADED requires the base tier. |
OIC | Oracle Integration Cloud — could serve as an orchestration layer (file transfer, callbacks, email) around the pipeline; not currently used. |
1 · Canonical object list (SSOT) VERIFIED 2026-07-05
One object = one upload unit — one FBDI ZIP, one HDL .dat submission, or one FBL ZIP, submitted to one Fusion import job. Sub-tables inside a ZIP are sub-objects (record types within the same FBDI), never separate objects. 45 in-scope objects (23 FBDI + 14 HDL + 6 FBL + 2 REST-loaded), plus PlanningBudgets — out of scope as of 2026-07-07. The tables below are the requirement — every object with its upload files, staging/transform tables, generated files, partitioning, and four registration checks. An object's code must be registered, spelled identically, in four places for the pipeline to work end to end: the scheduler sequence (so runs include it), the ERP options table (so the loader knows which Fusion import job to submit), the BIP registry (so reconciliation knows which report to run), and the display catalog view (so the UI can name it and drill into it). These places find each other by matching the code as text — a ✓ means that place has the object under the exact canonical code; a bold entry means it is spelled differently there, the lookup silently misses, and it must be fixed.

HDL objects (14)

2 · Execution engine (queue & heartbeat) VERIFIED 2026-07-04
How queued work actually gets executed. For what each pipeline stage does to the data, see Pipeline stages defined in the overview; for the status vocabularies, the status tables there.
The heartbeat
One permanent DBMS_SCHEDULER job, DMT_QUEUE_POLLER, fires DMT_QUEUE_PKG.HEARTBEAT_TICKsrc every 60 seconds (C_POLL_INTERVAL). Each tick is deliberately lightweight:
Promotes PENDING → READY when a work item's DEPENDS_ON predecessors are DONE — or, when the run was submitted with On Failure = Continue, when they are terminal (DONE or FAILED): dependents launch anyway and per-row dependency validation sorts out individual rows (decided 2026-07-07).
Handles data-dependent splits — partitions that can only be known after transform (Assets per book) get their per-partition work items created mid-run, and the Plan preview marks them "split at run". Partitions knowable from staging data (PO by business unit, GL by ledger) are computed by the Plan step and created at Confirm Submit — so "creates precisely what is shown" holds (stated 2026-07-07; per DMT_CEMLI_SPLIT_CFG).
Checks ESS state for items in AWAITING_LOAD / AWAITING_IMPORT / AWAITING_POSTRUN.
Spawns one-shot child jobs for the heavy work (below).
Rolls the run status up from its work items — the mapping is exactly the run-status definitions in the Overview's status table: each status's stated meaning is its rollup condition — defined once, there. Precedence (stated 2026-07-07): a run is IN_PROGRESS while any work item is unfinished — failures do not settle the run status early — and the terminal statuses apply only once every item is terminal.
Concurrency rule (decided 2026-07-07): an object may be part of only one active run at a time. Enforced at submission — a run whose selection includes an object already in an active run is rejected with a clear message naming the object and the blocking run. This removes row-level race conditions between overlapping runs by construction; background-job limits are the database scheduler's concern.
Child jobs — the heavy work
Heavy work runs in one-shot DBMS_SCHEDULER child jobs in a separate package, DMT_QUEUE_WORKER_PKGsrc (separate so the heartbeat never self-deadlocks on its own package in the library cache):
Procedure | Does |
EXECUTE_ONE | Runs the item's data phase: pre-validate → transform → post-validate → generate → submit load (the PROCESSING work status — currently named VALIDATING in code, P2 §12). |
POLL_ONE | One ESS check; advances the work item through the AWAITING_* states. Transient SOAP faults retry next tick; EXPIRED is not treated as terminal. |
RECONCILE_ONE | Runs BIP reconciliation and settles every row LOADED / FAILED. |
The worker calls DMT_LOADER_PKG.RUN_* with g_async_mode=TRUE (return-after-submit instead of the loader's own inline poll loop).
HDL work items use a subset of the statuses (decided 2026-07-07): PROCESSING (validate → transform → generate the .dat) → LOADING (upload + submit, one REST call each) → AWAITING_LOAD (poll the dataset to a terminal state; EXPIRED retries next tick) → RECONCILING (HDL failure messages + the object's Contract v1 report) → DONE or FAILED by the accounting rule. AWAITING_IMPORT and AWAITING_POSTRUN are never used by HDL items — HDL is one combined job with no post-run step.
REST-loaded work items (Banks, ARReceipts) use the shortest subset (decided 2026-07-07): PROCESSING (validate → transform) → LOADING (the record-at-a-time API calls, each recording its outcome as it goes) → DONE or FAILED by the accounting rule. No AWAITING states at all — there is no job to await; the load is the reconciliation.
Timeouts (decided 2026-07-07)
The polling timeout is configurable — ESS_POLL_TIMEOUT_MINUTES in
DMT_CONFIG_TBL (default 30), maintained at the administrator level: raise it before a 100,000-row load,
keep it short for everyday runs. A timeout is a trigger for the failure path, never a verdict: when it
expires, every GENERATED row of the file is marked FAILED with a [LOAD_ERROR]-tagged timeout message (the
all-or-nothing rule — an unfinished load committed nothing trustworthy), and reconciliation still runs —
if the job actually completed after we stopped watching, reconciliation finds the records in Fusion and marks them LOADED,
overriding the precaution. The work item ends DONE or FAILED by the accounting rule alone, never directly from the timer. A row rescued this way is LOADED with historical error text — a defined, legal state: status is always current, error text is the accumulated history (the override appends a confirmation note rather than erasing the timeout message), and the UNRECONCILED derivation is unaffected (it reads FAILED + empty text).
Human guidance, not engine behavior: a job still in WAIT/RETRY past ~20 minutes almost
always means a bad ParameterList — an operator (or an investigating session) should stop watching and check the parameters
rather than wait out the timeout. The engine never reads this number; nothing is marked differently because of it.

How they join
Worked example: a P2P run's actual rows — run, work queue, FBDI CSV/ZIP, TFM lineage →
PIPELINE_RUN 1—N WORK_QUEUE (by RUN_ID); each queue row → one FBDI_ZIP; each ZIP → one FBDI_CSV row per physical file in the archive (a PurchaseOrders ZIP has four); TFM.FBDI_CSV_ID→FBDI_CSV and TFM.STG_SEQUENCE_ID→STG. 5 · Error handling VERIFIED 2026-07-04
How failures are captured, attributed, and prevented from being mistaken for success.
Prefix convention. ERROR_TEXT is tagged by the stage that raised it — see the tag table below for when each is written and by which package group.
Accumulate, never overwrite. ERROR_TEXT is a CLOB appended via DMT_UTIL_PKG.APPEND_ERRORsrc at every step.
Per-row attribution. Errors attach to the specific header/line/distribution row that failed — no rolling everything up to the header with a generic "parent failed" cascade.
Cascade to children. A FAILED parent (e.g. PO header) cascades to its children (lines/locs/dists), each row carrying why.
absence ≠ LOADED. 0 BIP rows is never treated as success — it is investigated. A row that comes back with neither positive Fusion id nor an error is UNRECONCILED — a derived classification (FAILED + no ERROR_TEXT), not a stored status; see the status vocabulary in the Overview.
Object-status accounting. An object is DONE iff every record is accounted for (base-loaded OR interface-errored); FAILED only if any record is unaccounted. A Load ESS ERROR marks all GENERATED rows FAILED and routes to reconcile — it never short-circuits.
How each row's outcome is resolved (end state)
After the load ESS job finishes, reconciliation (DMT_{obj}_RESULTS_PKG.RECONCILE_BATCH) decides every TFM row's fate. In priority order:
Positive base-table match → LOADED, no error. The BIP report finds the row in the Fusion base table (Tier 2) with its Fusion ID. Nothing is appended to ERROR_TEXT.
Interface error table → FAILED + [FUSION_ERROR]. The row is still in the Fusion interface table with rejection text (Tier 1); the BIP report returns that text and it is appended per row. This is the standard path for FBDI objects (AP, PO, AR, GL, suppliers, customers, …).
Purging imports → FAILED + [IMPORT_REPORT] (fallback). The Projects-family imports (PJC_TXN_XFACE_STAGE_ALL: Projects, Expenditures) purge rejected rows from the interface table, so BIP can return 0 rows even when rows failed. The reconciler then parses the ESS import report via DMT_IMPORT_REPORT_PKG.PARSE_ERRORSsrc and matches errors back to TFM rows on the row identifier (ORIG_TRANSACTION_REFERENCE).
REST-loaded objects (Banks, ARReceipts) → outcome from each API response — success with the Fusion id, or a specific error, per record, recorded directly on the TFM row; no BIP step (low-volume master data).
HDL objects (HCM) → failures from the HDL response, positive success from a Contract v1 BIP report (decided 2026-07-07). The failure tier is free and per-record: every failing line comes back as a message identifying the record by SourceReference001 — the object's business key, the same value as the report contract's RECORD_KEY — and is marked FAILED with [FUSION_ERROR]. "Dataset finished and the record is not in the failure list" is only the interface-tier signal — never sufficient for LOADED on its own. LOADED requires the base-tier proof: a Contract v1 reconciliation report per HDL object returns every record's key and Fusion-assigned id from the HCM base tables in one bulk call (a per-record REST lookup was rejected: at conversion volume — tens of thousands of workers — one call per record takes hours; the partial 3-object REST lookup retires). Records that are absent from the failure list but also missing from the report fall to the [RECONCILE_ERROR] sweep.
Still unaccounted → FAILED + [RECONCILE_ERROR]. The unmatched-row sweep: not found in the BIP result, base table, or import report — "cannot verify import outcome." A row left FAILED with no ERROR_TEXT surfaces as the derived UNRECONCILED classification (Overview) and fails the object's accounting rule. End state — this sweep is enforcement, not per-object code: it runs as a shared common utility invoked by DMT_QUEUE_WORKER_PKG.RECONCILE_ONEsrc after each object's RECONCILE_BATCH, driven by the TFM-table catalog, so no reconciler can omit it. Today it exists inline in only 9 of ~42 reconcilers (P2, §12).
When a load fails, assume our code or our data is wrong — download the Fusion job's diagnostics first and find the defect on our side; "the instance is broken" is the last hypothesis, not the first.
Which columns hold what (stated to remove ambiguity): the outcome word (LOADED / FAILED) goes in the TFM row's STATUS column, and only there. The Fusion identifier goes in the object's FUSION_*_ID column(s) (e.g. FUSION_INVOICE_ID) — IDs only, never words: a FUSION_*_ID column never contains "LOADED" or any other text. Rejection text goes in ERROR_TEXT.
Independently of row status, DMT_LOADER_PKGsrc calls DMT_IMPORT_REPORT_PKG.PARSE_AND_LOG_ERRORSsrc after each FBDI load to write all import-report errors to DMT_LOG_TBL — run-level diagnostics only; it does not touch TFM rows.

Volume — keyset pagination (decided 2026-07-07). Every data model orders its rows by RECORD_KEY and returns at most P_CHUNK_SIZE rows after P_AFTER_KEY. The shared fetch function loops — first call with an empty cursor, each next call passing the last key received — feeding each page to the parser as it arrives, until a page returns fewer rows than the chunk size. Memory is bounded to one chunk regardless of total volume; a page-count cap derived from the run's generated-row count stops a misbehaving report from looping forever. Keyset was chosen over offset-plus-overlap paging: the cursor is exact (no gaps, no duplicates, no drift — and the reconciliation population is fixed anyway once the load and import jobs finish), and every page costs the same. A scheduled bulk file export was considered and rejected as heavier machinery (delivery infrastructure, job polling, giant-file parsing) for the same result.
Outcome semantics. A TFM row is LOADED only from a BASE / SUCCESS / FUSION_ID NOT NULL
row; INTERFACE/SUCCESS is corroborating, never sufficient. Zero report rows is never success — the reconciler logs a warning,
rows stay unaccounted, and the accounting rule fails the work item. A SOAP fault raises immediately, never a silent retry.
HDL objects conform too (decided 2026-07-07): each HDL object gets a Contract v1 report whose rows are the BASE tier (key + Fusion id) — its ERROR rows are optional, because HDL failures arrive per-record in the HDL response and are tagged [FUSION_ERROR] before the report runs. The report's RECORD_KEY is the same business key written into the .dat as SourceReference001 — one key definition per object serving the screens, BIP matching, and HDL matching.
One generic parser. The ~25 bespoke parse blocks collapse into one shared parser reading only the seven
standard columns, driven by per-object registration added to DMT_BIP_REPORT_TBL: TFM_TABLE,
FUSION_ID_COLUMN, and CONTRACT_VERSION — the key-building expression (RECON_KEY_SQL) lives on the per-object registry, shared by every consumer (decided 2026-07-07) (1 = conforms; 0 = legacy — the
flag lets standard and legacy coexist during per-object migration, each step gated by the regression scenario).
Showing final outcomes next to staging data (decided 2026-07-07): screens that want a "Loaded" column on staging lists read a view joining each staging row to its most recent TFM attempt — computed at query time from the single source of truth. Activity log — attribution contract QUEUE_ID + context: TO-DO
DMT_LOG_TBL is the execution log for every package: LOG_ID (sequence — the read order), LOG_DATE, LOG_TYPE (INFO milestone / WARN non-fatal / ERROR with SQLERRM_TEXT), PACKAGE_NAME + PROCEDURE_NAME (who wrote it), MESSAGE (CLOB — full context fits), and the attribution ids below. It is written via autonomous transactions, so log entries survive rollbacks — when a step fails and its data changes roll back, the entries saying why still commit.
Attribution ids (decided 2026-07-06). Every entry carries RUN_ID (exists) and QUEUE_ID — the work item that wrote it (to add). Both nullable: NULL means "no run/work-item context exists" (heartbeat ticks, config operations, entries preceding the run row). Both indexed. No foreign keys, by design — log rows may precede the run row, and diagnostics must never fail to write because of referential ordering.
Coding standard — session log context. Procedures do not pass ids for logging. DMT_QUEUE_WORKER_PKGsrc calls DMT_UTIL_PKG.SET_LOG_CONTEXT(p_run_id, p_queue_id) at the top of each child job (EXECUTE_ONE / POLL_ONE / RECONCILE_ONE); standalone entry points set run-only context; DMT_UTIL_PKG.LOGsrc stamps both ids on every entry automatically. Only tick-level heartbeat entries log NULL. (Also to be added to docs/coding-standards.md.)
Reading it (§9): one Activity Log browser screen, reached pre-filtered from Run Detail (this run) and Object Detail (this work item) — the Dashboard has no log entry point (decided 2026-07-06). Filtering by object joins the entry's work-item id through the work queue to the object code; entries with no work-item id (heartbeat ticks) never match an object filter, by design (decided 2026-07-07).
Proposed — run-stamped home for pre-TFM failures TO-DO
Decided (2026-07-06, refined 2026-07-07) — errors only; success is the TFM row. The table below records a row per failed STG record per run attempt — errors only (decided 2026-07-07): reaching the TFM table is the success record, since the TFM row carries the run id and a created timestamp. Together, this error table and the TFM table are the complete per-run record of what happened at the stage→transform step. The STG row carries the forward-only status (NEW → TRANSFORMED or FAILED — final decision 2026-07-07), written only by the stage→transform step, and never a message; failures live in this error table and success is denoted by the TFM row itself (run-stamped, timestamped) — no explicit success rows (decided 2026-07-07). The FAILED-mode "clearing" contradiction (section 13) dissolves — there is nothing on the STG row to clear — and run-mode selection (NEW / FAILED / ALL) becomes predicates over this table and TFM, not over an STG status column.  Downstream impacts are listed with the decision in section 13.
Rows that fail before reaching TFM (pre-validation / transform) have no run-stamped home today: STG has no RUN_ID, so these failures are only flagged on the STG row and cannot be attributed to a historical run. The table below closes this gap; the per-run funnel that consumes it is defined in §10.
New table — transform errors only
Downstream errors already live on the run-stamped TFM row. The only gap is rows that never reached TFM. So the new table is scoped tightly to that — no TFM_SEQUENCE_ID (these rows have no TFM row by definition):
DMT_STG_TFM_ERROR_TBL
ERROR_ID         PK
RUN_ID           -- the run that rejected it  (gives historical attribution STG lacks)
QUEUE_ID         -- the work item that rejected it (nullable; decided 2026-07-06)
CEMLI_CODE       -- e.g. PurchaseOrders
SUB_OBJECT       -- e.g. PO Lines  (identifies which STG table, for multi-CSV FBDIs)
STG_SEQUENCE_ID  -- FK to the staging row that failed to transform
ERROR_TEXT       -- why it failed
CREATED_DATE
Wiring: when a pre-transform validation or transform rejects a staging row, INSERT it here with the RUN_ID — in addition to flagging the STG row FAILED (the STG flag stays; FAILED-mode reruns select on it). Pre-validation runs inside the work item, so both ids are always known when the row is written; for partitioned objects the row's partition is derived from its partition column(s) like any other row (section 9). ERROR_TEXT carries the §5 tags ([PRE_VALIDATION] or [TRANSFORM_ERROR]) so this table and the TFM rows share one tag vocabulary. Build the metrics into a view, and roll out one pipeline end-to-end (Suppliers) before generalizing.
With this table the earlier idea of a "two-number run stamp" is unnecessary: Staged becomes derivable as Transformed + Transform-errors, both run-stamped.

Credentials (decided 2026-07-07)
How a credential is resolved — master with per-object overrides. One master Fusion account lives in
DMT_CONFIG_TBL and is the default for every Fusion call. Individual objects override it where a load or a BIP report call needs different permissions — every object's override lives in one place: a credential-override column on the object registry (the one-row-per-object registration table the catalog-driven dispatch reads — decided 2026-07-07; the existing FBDI override rows in the ERP interface options migrate there, so FBDI, HDL, and REST-loaded objects all resolve identically). The resolution rule is: use the object's override if one exists, otherwise the master. One consequence is a hard
rule learned in production: job-status polling must use the same account that submitted the job — Fusion
will not show one user's job to another, so the poller resolves credentials exactly the way the submitter did.
Inventory: every credential the tool holds (master account, per-object overrides, wallet) is listed
with its storage location — no credential exists that isn't in the inventory.
Display: the Administration screen shows credentials masked, with an explicit reveal action — never
in plain lists or exports.
Rotation: one procedure updates every place a credential is cached, in one operation (this exists
operationally today); partial rotation is how loads break with 401 errors.
Never in git: database exports, seed files, and repository content never contain credential values —
the deploy snapshot masks them, and a committed credential is a defect (one was found and is tracked as issue #36).

7 · Coding standards VERIFIED 2026-07-06
Rules every DMT package, view, and page region follows. Initial set from the requirements review plus the 2026-07-06 audit of the MCCS conversion codebase (the predecessor project — its conventions and its scars both inform these). To be mirrored in docs/coding-standards.md.
Gold standard — worked example. The complete AP Invoices object (DMT_GOLD_STANDARD.html (rendered page: README + full source of all 8 files; raw files in docs/gold-standard/): validator, transform, FBDI generator, reconciler) is written to every standard below and is the model for all other objects. Its README maps each standard to file:line examples. Reference code — not yet deployed; it codes against the end-state contracts (SET_LOG_CONTEXT, ERR$_ error tables) and its assumptions are listed in the README.
Standard | Rule | Why | Known violations to sweep |
No EXECUTE IMMEDIATE | All runtime SQL is static SQL. Dynamic SQL is banned in packages, page regions, and views. | Dynamic SQL defeats compile-time checking and ALL_DEPENDENCIES — the exact trap that made the Object Detail wiring undiscoverable (§11's investigation); it also hides table usage from impact analysis and invites injection. | Object Detail breakdown region (dynamic per-table COUNTs), DMT_LOADER_PKG.reset_scenario_statussrc + funnel-classification query, ESS-id lookup on Object Detail. Approved exception (2026-07-07), scoped hard: dynamic SQL is permitted only in deploy scripts — the guarded idempotent DDL blocks in schema/ files that run at install time and cannot be written statically. It may never appear in any code object that lives on the database: no package, procedure, view, trigger, or page region. If a database object needs dynamic SQL, the design is wrong. |
No LIKE on known codes | Object codes, statuses, and lookup codes are a controlled vocabulary (§1) — match them with = or IN, never LIKE '%...%'. | Pattern matching on a controlled value is a latent routing bug: it matches values it was never meant to and silently misroutes new ones. The live example: 1099 routing on INVOICE_TYPE_LOOKUP_CODE LIKE '%1099%' flows into the FBDI as-is (§1 defect). | 1099 routing filter (loader/validator/transform/generator); audit for others. |
Type suffix at the end | Object-type designator is always the suffix: _TBL, _V, _PKG, _SEQ. Never a prefix or mid-name. | One predictable pattern makes names sortable, greppable, and self-describing; two conventions guarantee misses in searches and sweeps. | The DMT_V_* view family (DMT_V_CEMLI_TFM_TABLESsrc, DMT_V_CEMLI_STATUSsrc, …) → rename to DMT_..._V; fold into the naming conformance sweep (§12). |
One BEGIN/END per procedure | No mid-procedure anonymous BEGIN…END blocks. A procedure or function has exactly one BEGIN…END; if a piece of logic needs its own block or its own exception scope, it becomes its own (private) procedure. | Nested blocks hide control flow and let per-block WHEN OTHERS handlers swallow errors mid-stream; they are also where copy-paste logic accumulates. The 2026-07 code review flagged this pattern across the codebase. | Multiple packages (2026-07 review, issue set #43); sweep during refactors. |
Every procedure returns an error code | Every procedure catches its own errors and reports outcome to the caller through an IN/OUT error-code parameter (0 = success, non-zero = failure; message text goes to the §5 log / ERROR_TEXT paths). Exceptions never escape a procedure unhandled. | Callers get a uniform, checkable contract instead of guessing which exceptions might propagate; failure handling becomes explicit at every call site — which is what the §5 accounting rule depends on. | Much of the current code signals by unhandled exceptions; applies to all new code immediately, retrofit folded into refactor sweeps (§12). |
Identifier prefixes — one set | Parameters and variables use exactly these prefixes: p_ IN parameter, x_ OUT / IN OUT parameter, l_ local variable, g_ package-level state, c_ constant. Nothing else. (MCCS mixed five schemes — pv_/gv_/gi_/cv_/bare — so any search for "all package state" or "all OUT params" needed five patterns and missed silently.)
✗ pv_job_id · gv_debug_mode · v_error_msg · p_message OUT
✓ g_job_id · g_debug_mode · l_error_msg · x_message OUT
|
VARCHAR2 always; flags are VARCHAR2(1) | Never VARCHAR; never CHAR for Y/N flags — CHAR blank-pads, so flag = 'Y' comparisons fail unpredictably against padded values.

What a tile counts, and partitions (decided 2026-07-06)
Tile numbers count parent records only — for PurchaseOrders, orders (headers), not lines. But a parent is not successful unless all of its children loaded: an order with a failed line counts as failed, so child failures always surface in the tile numbers and color. The per-record-type split lives on the Object Detail funnel.
The rollup formula — worst child wins (decided 2026-07-07): a parent record's displayed state is the worst state across itself and all of its children, ranked unaccounted > failed > loaded. A fully clean order counts as loaded; an order with any failed line (all accounted) counts as failed; an order with any unexplained line counts as unaccounted. The three counts always sum to the parent total, and the tile palette applies to these rolled-up numbers unchanged.
Partitioned objects render one tile per partition (PurchaseOrders per Procurement BU, GL Balances per ledger) — each with its own counts and color. This is 1:1 with work items, which are already created per partition, so the tile palette stays honest per work item; the drill passes the partition key alongside the object code.
Tile colors on Run Detail (decided 2026-07-06)
Color |  | Meaning |
Green |  | 100% of records loaded — every record confirmed in Fusion base tables. |
Light green |  | Completed with some failures — every record accounted for, some ended FAILED with a reportable error (COMPLETED_ERRORS). |
Light red |  | All records failed, or some records are unaccounted (no Fusion ID and no error). |
Red |  | Infrastructure failure — uncaught exception, errored loader, BIP report failure, or all records unaccounted. The pipeline itself broke; record outcomes are unknown or untrustworthy. |
Grey |  | No rows to process (nothing selected, or skipped because a dependency failed). |
Blue |  | In progress. |
White |  | Queued, not started. |
The light red / red distinction is deliberate: light red means the pipeline worked and the data was bad (rows failed with reportable errors, or a few slipped through unaccounted); red means the pipeline itself broke — an exception, a dead loader, a BIP report that would not run, or every record unaccounted (the signature of a systemic failure, not bad data).
Gap — current app does not implement this palette (P2, §12). Today dmt_run_detail_tiles.sql colors by status string on two diverging code paths: the queue path shows only green/red/grey/white/blue (a mixed loaded/failed object shows plain green), and the sync path shows COMPLETED_ERRORS as orange. Neither distinguishes data failures from infrastructure failures, and unaccounted records do not influence color at all. Implementing this requires the per-object outcome counts (loaded / failed-with-error / unaccounted — the same numbers the §10 funnel uses).

How rows belong to a partition (stated 2026-07-07). Nothing extra is stamped: a row's partition is derived by evaluating the object's partition column(s) — defined in DMT_CEMLI_SPLIT_CFG (e.g. the business unit on a PO row, the ledger on a GL row) — directly on the TFM or error-table row. Every partition-scoped count, list, and log filter applies that same predicate, which is why a partition tile's count and its drilled record list can never disagree.
One business key per object, defined once. The Record List's "Key" column and the reconciliation
report contract's RECORD_KEY (section 5) are the same concept: the one value that identifies a record of
that object (supplier name, PO number, project number…). Each object defines its key exactly once — as the key-building expression on its row in the per-object registry (decided 2026-07-07; the same table holding the run procedure and credential override). At generate time the transform stamps the computed value onto each TFM row; from then on the reconciliation report returns it as RECORD_KEY, the HDL generator writes it as SourceReference001, the Record List displays it, and the parser matches on plain equality. Two definitions of the
same key is how a screen ends up showing a key the reconciliation cannot match.
10 · Object Detail & the funnel UI VERIFIED 2026-07-06 redesign TO-DO
Page 52's "Record Breakdown" is where the funnel lives — today it misleads, the redesign makes it the pipeline funnel.
Current behaviour — and why it misleads
The breakdown is an inline PL/SQL region on the Object Detail page: it reads the object display catalog view DMT_V_CEMLI_TFM_TABLESsrc (§11) and counts each record type's TFM table by current STATUS, drawing Sub-Object | Staged | Generated | Loaded | Failed | Total, each count a clickable pill to the Record List. Because row status is overwritten as a row advances (and never reset), a completed run has every row terminal ⇒ Staged = 0 and Generated = 0 by construction — the columns look like a funnel but cannot be one. Maintenance warning: DMT_APEX_PAGE_PKG.RENDER_OBJECT_BREAKDOWNsrc looks like this screen's renderer but is un-wired dead code, and DMT_OBJECT_DETAIL_Vsrc backs only the deprecated Object Detail v2 page plus a one-row job-id lookup — editing either does not change this screen. §11 records the verified wiring.
The funnel — definition TO-DO
Why the current counts can never be a funnel: STATUS is one mutable column, overwritten as a row advances — and pre-TFM failures need the run-stamped error table (§5) to be countable for historical runs. The funnel therefore takes each stage from a durable source:
The funnel (stages 1, 3–6 are cumulative — each ⊇ the next; row 2 is the drop-out between 1 and 3)
# | Column | Definition | Source |
1 | Staged / available | Rows in STG for the run's scenario | = Transformed + Transform-errors (derived; see below) |
2 | Transform errors | Failed validation/transform — never reached TFM | DMT_STG_TFM_ERROR_TBL count for the RUN_ID |
3 | Transformed | Landed in the TFM table for this run | COUNT(TFM WHERE RUN_ID=x) |
4 | Generated (in FBDI) | Written into the FBDI/HDL file, ready to push | COUNT(TFM WHERE RUN_ID=x AND FBDI_CSV_ID NOT NULL) |
5 | Interfaced | Rows that physically arrived in Fusion | All-or-nothing per ZIP (decided 2026-07-07): the load unit is the ZIP — one load job per ZIP, and if any CSV inside it fails, the whole ZIP is rolled back; nothing partial survives. If the ZIP's load job truly succeeded, every Generated row of that file is Interfaced; if it failed — including a WARNING, which means zero rows committed — none are, and every row carries [LOAD_ERROR]. So per file: Interfaced = Generated on load success, 0 on load failure. Never inferred from "the job finished" — the job's actual outcome decides. |
6 | Loaded / App errors / Unaccounted | Confirmed in Fusion base tables / rejected at import / neither | TFM STATUS='LOADED' / FAILED with [FUSION_ERROR] or [IMPORT_REPORT] / the derived UNRECONCILED bucket (Overview, §5) |
HDL and REST-loaded objects (decided 2026-07-07): for HDL, Interfaced = the dataset accepted the submission — all-or-nothing per .dat, exactly like the ZIP rule — and the per-record failures then land at the Fusion stage from the dataset messages. REST-loaded objects have no Interfaced stage at all: the funnel collapses to Staged → Transformed → Loaded/Failed, because each API call is simultaneously the load and the reconciliation.
Total errors = Staged − Loaded − Unaccounted, decomposed into the drop at each stage. Unaccounted is surfaced as its own number, never folded into an error column — it is the count the accounting rule exists to drive to zero.
Mock-ups: the funnel on Object Detail — the horizontal per-record-type funnel table with drop-out counts, and the row expansion showing one record type's funnel visually. One image, one place: the mock-ups page.
11 · Object display catalog — how the UI names and drills each object VERIFIED 2026-07-06
Every drill-through screen needs three facts about each object: what to call it on screen, which TFM table holds its records, and how to build the drill link to those records. That mapping is the object display catalog (the DMT_V_CEMLI_TFM_TABLESsrc and DMT_RECORD_DETAIL_Vsrc views). When the catalog and the pipeline disagree — a code spelled two ways, a label that drifted, a missing branch — a screen shows the wrong counts or a dead drill link. This section defines the catalog and records the audit that found and fixed those defects.
Canonical values every layer must use
Screens match these names as exact text — the display name a drill link carries and the name the target screen filters on must be identical to the byte; a drifted label lands the user on an empty list (the defect class recorded in the Appendix).
Code spelling. The canonical code for every object is decided per object in the canonical object list (section 1) — neither the long nor the short form wins globally (GLBudgets was decided short; PayrollRelationships long). Every layer — scheduler, queue, generators, reconcilers, this catalog — conforms to section 1's decided code; the remaining mismatches are the naming conformance sweep in section 12.

How the catalog is maintained (decided 2026-07-07). The catalog's entries move out of the hardcoded view (today ~90 UNION ALL … FROM DUAL branches) into a normal seed table — DMT_CEMLI_CATALOG_TBL: one row per record type per object carrying object code, display name, transform table, status column, row filter, and sort order — and DMT_V_CEMLI_TFM_TABLES becomes a plain SELECT over it. Registering an object is then three seed inserts — four for FBDI objects, which also need their ERP-options row: its pipeline definition rows (section 6), its BIP registry row (section 6), and its catalog rows here. The catalog-driven queue dispatch (section 12 backlog) reads the same registration data.
How the catalog is enforced
Catalog health is guarded automatically: the DMT_V_CATALOG_HEALTHsrc validator view checks that every catalog entry names a real table, column, and filter, and the regression harness (scripts/dmt_run_assert.py) asserts on every gate run that the catalog, the record views, and the live screens agree (drill pairs resolve, one row per work item, no unknown codes). The lettered assertions the Appendix cites (A–E) are enumerated in that script — the script is their definition; the letters here are citations, not spec. A catalog defect fails the regression gate rather than waiting for a user to hit a wrong count or dead link.

An independent reviewer read this document with no other context and listed everything a new person would have to ask a human before building or approving. Q1 — What do a Run Detail tile's numbers count? DECIDED 2026-07-06
Decision: parent records only — but a parent is not successful unless all of its children loaded. Written into section 9.
The Run Detail screen shows one tile per object, e.g. "PurchaseOrders — 104 loaded / 15 failed". But one PurchaseOrders FBDI contains four record types with their own counts (headers 28 loaded, lines 104, locations 110, distributions 127). The document never says whether the tile shows the sum of all record types (369 loaded), one record type, or something else. The tile color rules (section 9) depend on these numbers, so the palette cannot be implemented until this is decided.
Options: (a) sum every record type — "369 loaded / 15 failed / 1 unaccounted" — honest but big numbers; (b) show only the parent record type (headers: "28 / 1") — small numbers but hides child failures; (c) sum for the color decision but display "records: 369 ✓ · 15 ✗ · 1 ?" with the per-type split on the Object Detail funnel. Suggestion: (c) — color must consider every record (a child failure must be able to turn the tile light green/red), and the funnel is where the split already lives.
Q2 — How do partitioned objects appear on screen? DECIDED 2026-07-06
Decision: one tile per partition, each with its own counts — 1:1 with work items, which are already created per partition (so each partition has its own work-queue id today; no new mechanism needed). Written into section 9.
A grouped object splits into one work item per partition — PurchaseOrders per Procurement BU, GL Balances per ledger (config table DMT_CEMLI_SPLIT_CFG). Each partition gets its own ZIP and its own ESS jobs. But the Object Detail screen receives only a run id and an object code — no partition. So: one tile per object or one per partition? Whose ESS job chips does Object Detail show? Does the funnel sum across partitions?
Options: (a) one tile per partition ("PurchaseOrders — US1", "PurchaseOrders — US2") — matches the work-item granularity exactly, so the tile palette is honest per work item, and Object Detail receives the partition key and shows that partition's jobs and funnel; (b) one tile per object, aggregated, with a partition list inside Object Detail. Suggestion: (a) — the tile represents a work item everywhere else in the design; keeping that 1:1 avoids inventing an aggregation layer. Requires passing PARTITION_KEY alongside the object code in the drill.
Q3 — FAILED-mode re-runs clear ERROR_TEXT DIRECTION DECIDED 2026-07-06
Decision (updated 2026-07-07): the stage→transform result table records failures only — success is the TFM row itself (run-stamped, timestamped) — and the STG row carries a forward-only status (NEW → TRANSFORMED or FAILED, reflecting the most recent stage→transform attempt, written only by that step — final decision 2026-07-07). Messages never go on staging; downstream outcomes are never written back to staging; FAILED-mode selection combines staging FAILED rows with rows whose most recent TFM attempt failed; dependency checks read the parent's TFM row. Nothing to clear, full history per run. Written into section 5. Downstream impacts to accept: (1) STG ERROR_TEXT retires — any screen showing a staging row's error must join the outcome table for the latest attempt instead; (2) volume — one row per failed record per attempt; covered by the retention policy (section 6), and STG/TFM stay lean; (3) downstream outcomes are never written back to staging — staging status is written only by the stage→transform step; (4) the name stands as-is.
The Overview run-mode table says FAILED mode re-attempts failed rows and "ERROR_TEXT is cleared". Error handling (section 5) says ERROR_TEXT "accumulates, never overwritten". The status vocabulary calls the ALL-mode reset "the one sanctioned status reset". The code really does clear (supplier transform):
IF p_reprocess_errors THEN
UPDATE DMT_POZ_SUPPLIERS_STG_TBL
SET    ERROR_TEXT = NULL, LAST_UPDATED_DATE = SYSDATE
WHERE  STATUS IN ('FAILED', 'TRANSFORM_FAILED');
END IF;
Options: (a) declare FAILED-mode clearing the second sanctioned reset — the STG row's ERROR_TEXT describes the current attempt only, and history is preserved anyway: every attempt's errors are in the activity log, and (once built) the run-stamped error table keeps pre-transform failures per run; (b) never clear — append an attempt separator instead ("—— retry, run 117 ——"), so one column holds the full history at the cost of ever-growing CLOBs and confusing "is this error current?" reads. Suggestion: (a), with the Overview and section 5 reworded to say exactly that: current-attempt errors on the row; full history in the log and the run-stamped error table.
Q4 — Where are pipelines defined? DECIDED 2026-07-06
Decision: the pipeline definition table is added — see DMT_PIPELINE_DEF_TBL in section 6, including the post-run job column (Q7) and, as of 2026-07-07, the complete seed content (membership + dependency graph for every in-scope object).
P2P, O2C, FIN and the rest are used everywhere ("resolves the pipeline code(s) to a CEMLI sequence"), and section 1 makes "registered in the scheduler sequence" one of the four mandatory checks — but no table or file holding pipeline definitions is ever named. In the code today the sequences are hardcoded inside DMT_SCHEDULER_PKGsrc, which means adding an object to a pipeline is a package edit.
Suggestion: a pipeline definition table, seeded like the other control tables — this is the same decision already made for queue dispatch ("catalog-driven dispatch", P2 in the backlog); doing both from one catalog table kills two hardcodings at once:

Q5 — The prefix is never actually specified DECIDED 2026-07-06
Decision: 4-digit prefixes from one sequence starting at 1000 (no test/production ranges — an earlier draft's "9001+" convention was wrong), prefixing forced at submission — no user choice; production cutover only via the administrator-controlled USE_PREFIX configuration value (NULL prefix on such runs). Prefixed key(s) per object to be documented as a section 1 column (P3). Written into section 6.
The Overview says a run stamps "a fresh numeric prefix onto business keys"; the mock-ups show 9627PO-1002. Not written down anywhere: where the numbers come from and their range; which business key(s) get prefixed for each object (PO number? supplier name and number? project number and name — the ProjectBudgets defect happened exactly because one reference wasn't prefixed); whether a production run stores NULL prefix; and how a user requests a no-prefix production run — no submission flag appears in the document.
Suggestion: a "Prefix" subsection in Control & config tables (section 6) stating: one sequence (DMT_RUN_PREFIX_SEQ) assigns every prefix; the prefixed key(s) per object are listed as a column in the canonical object list (section 1) — that makes it reviewable per object; submission takes p_use_prefix defaulting to Y, and a production cutover passes N storing NULL on the run row.
Q6 — What is the life of an HDL work item? DECIDED 2026-07-07
Decision: HDL items use the status subset PROCESSING → LOADING → AWAITING_LOAD → RECONCILING → DONE/FAILED (no import or post-run states — written into section 2). Failures are positive and per-record from the HDL messages; positive success requires the object's Contract v1 BIP report (bulk key + Fusion id from the HCM base tables) — the not-in-the-failure-list complement rule is never sufficient alone, and the per-record REST lookup was rejected on volume (hours at conversion scale). Written into section 5; build work in the backlog.
The work-item statuses are shaped around ESS jobs and BIP reconciliation (AWAITING_LOAD / AWAITING_IMPORT / RECONCILING). HDL objects load over REST and, per section 1, "HDL reconciliation isn't built yet". So which statuses does an HDL item pass through, what does DONE mean for the 14 HDL objects, and under the accounting rule (every record positively accounted for) can one ever legitimately reach DONE today?
Suggestion: define the HDL lifecycle as VALIDATING → LOADING (REST upload + submit) → AWAITING_LOAD (poll the dataset status) → RECONCILING (retrieve per-row results) → DONE/FAILED — the reconcile step already has a concrete mechanism to specify against, the HCM REST messages endpoint the utility package documents:
-- dmt_hdl_util_pkg.pks
--  Errors: GET /hcmRestApi/resources/11.13.18.05/dataLoadDataSets/{RequestId}/child/messages
And state plainly: until that reconcile step is built, an HDL work item may not be marked DONE — it ends FAILED with its rows unaccounted, which is the truthful outcome. That closes the loophole where an HDL "load succeeded" could be mistaken for records confirmed.
Q7 — Which objects have a post-run step? DECIDED 2026-07-06
Decision: post-run job is a column on the pipeline definition table (section 6). Today only Assets (per-book Prepare + Post); all others explicitly none.
The status exists ("post-run ESS step — per-object post-processing job submitted; polling") but no section says which objects have one or what the jobs are. From the code, Assets is the known case: after import, each asset book runs Prepare and Post jobs. Suggestion: add a "post-run job" column to the per-object catalog (the same table proposed in Q4) — today it holds a value only for Assets; every other object is explicitly "none", which is itself information.
Q8 — Rows rejected at the load step DECIDED 2026-07-06
Decision: the load step is all-or-nothing per file — a SQL*Loader failure errors the whole load job and nothing arrives; individual rows cannot fail this step alone. A [LOAD_ERROR] tag is added (section 5) applied to every GENERATED row of the failed file — never to individual rows. The funnel's per-row "rejected at load" drop is removed from the mock-ups.
The funnel shows a drop between Generated and Interfaced ("2 rejected at load") — rows that were in the file but never arrived in the interface tables (SQL*Loader rejected them: bad format, wrong data type, oversize value). Section 5's tag table has no tag for this: [FUSION_ERROR] is for interface-table rejections, [IMPORT_REPORT] is described only as the Projects-family fallback. Which tag, parsed from where, written by what?
Suggestion: a [LOAD_ERROR] tag, produced during reconciliation by parsing the load ESS job's log/rejection output (the same DMT_IMPORT_REPORT_PKGsrc parsing machinery pointed at the load job instead of the import job), matched to rows by position/key like the import-report fallback. Rows still unmatched after that fall to the [RECONCILE_ERROR] sweep as today.

Q11 — The cross-run dependency check and the dependent prefix don't compose into one algorithm
The situation, step by step. Suppliers were loaded in run 42, which used prefix 1042 — so the
supplier the source system calls "ACME Corp" exists in Fusion as "1042ACME Corp". Weeks later you run
PurchaseOrders as run 57 (its own prefix, 1057). Each PO's staging row references its supplier by the raw name,
"ACME Corp". Two things must now both work:
The dependency check — "is this PO's supplier loaded?" — must compare the values exactly as they appear in the source data — no prefix applied: the PO's
"ACME Corp" against the supplier staging row's "ACME Corp" with STATUS='LOADED'. Comparing the raw name to the
prefixed value ("1042ACME Corp") can never match — that was exactly the ProjectBudgets defect fixed 2026-07-03.
The reference written into the FBDI file must carry the supplier run's prefix, not the PO run's:
the PO file must say supplier "1042ACME Corp" (what Fusion actually contains), not "ACME Corp" (doesn't exist
in Fusion) and not "1057ACME Corp" (wrong run's prefix — also doesn't exist).
-- 1. VALIDATE (source values as-is, no prefix, against the parent's STAGING row):
SELECT 1 FROM dmt_poz_suppliers_stg_tbl s
JOIN  dmt_poz_suppliers_tfm_tbl t ON t.stg_sequence_id = s.stg_sequence_id
WHERE s.supplier_name = po.supplier_name  -- 'ACME Corp' = 'ACME Corp' (source values, no prefix)
AND   t.status = 'LOADED';  -- loaded-ness is proven on the TFM row, never on staging
-- 2. TRANSFORM (write the PARENT run's prefix into the reference):
tfm.vendor_name := p_dependent_prefix || stg.supplier_name;  -- '1042' || 'ACME Corp'
-- 3. p_dependent_prefix ('1042') is picked at submission from the prefix-history view:
--   "which run loaded the suppliers I depend on, and what prefix did it use?"
DECIDED 2026-07-07: the pattern is the standard, generalized — the check accepts a LOADED parent from any run, the reference prefix comes from the parent's most recent LOADED row (the dependent-run picker is an optional override), and dependency validation can be turned off per run (submission screen, section 8).
Q12 — Retention DECIDED 2026-07-06
Decision: config-driven retention — written into section 6 (RETENTION_DAYS, content purged, metadata and STG/TFM never).
Every run stores full CSV content (CLOBs), full ZIPs (BLOBs), ESS output files, and an ever-growing activity log that survives rollbacks by design. No retention or purge policy exists anywhere. Suggestion: a config-driven purge (e.g. RETENTION_DAYS in DMT_CONFIG_TBL, default 90): a scheduled job empties CSV/ZIP/ESS-file content older than the window while keeping the metadata rows (lineage stays intact), and deletes log entries past the window. STG/TFM rows are never purged — they are the record of what was migrated.
