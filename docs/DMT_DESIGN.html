Purpose. The point of the tool is not to pre-validate all the data (although some validation is possible) — it is to move all available data into the cloud and ensure that every record is accounted for through a positive success or a positive failure message. The tool gives users one place to put data, and hides the complexity of running the loads, re-running the same data, and reporting on different loads that could otherwise load the same data twice.

DMT moves data from source systems (Oracle EBS or generic CSV) into Oracle Fusion Cloud through a multi-stage pipeline. Two load patterns: FBDI (CSV, ERP modules) and HDL (DAT, HCM). Everything is pure PL/SQL on ATP (schema DMT_OWNER); APEX is the UI layer.

Ways to get data in.

  • CSV upload with proprietary load syntax — users send in data shaped like the FBDI, but column order is determined by the file header rather than fixed position. IMPLEMENTED
  • FBDI structured upload — users upload files in the exact FBDI-generated format: position-based columns, no header row, as produced by the Oracle FBDI templates. Both this and the proprietary format are accepted by the same upload path. IMPLEMENTED
  • Direct DB load — insert straight into the STG tables (EBS DB-link push or SQL). The EBS push comes from the separate EBSAdaptors repo through a three-object interface contract — see §4. IMPLEMENTED
  • REST APIs exposing the STG tables for external loaders. TO-DO (future state)
  • AI-assisted loading — allow an AI agent to move data into the STG tables, or expose an MCP server over the STG layer. TO-DO (next stage after REST)

Pipeline stages defined

Worked example: follow two real records through every stage and status change →

StageWhat it doesTables touched
StageRaw source rows land in staging — via CSV upload (header-driven syntax), EBS DB-link push, or direct SQL. Rows arrive STATUS='NEW', tagged to their mandatory scenario. No prefix, no run association yet.DMT_{obj}_STG_TBL
Pre-validateRuns on STG rows before anything else. Two distinct checks:
(1) Cross-object dependency — has the master data this record needs (a different object, loaded in an earlier run) gone all the way into Fusion? Requires the upstream record to have a LOADED TFM row from any run (decided 2026-07-07) — within one run, the dependency graph guarantees the parent work item finished first, so just-loaded parents qualify; in a failed-items re-run, parents loaded in earlier runs qualify. The match compares the source values as they appear in the data (no prefix) on both staging sides, then joins to the parent's TFM outcome. The reference written into the load file carries the prefix of the parent's most recent LOADED row — automatically correct for same-run, prior-run, and mixed cases; the submission screen's dependent-run picker is an optional override to pin a specific run. This validation can be turned off per run ("Validate upstream dependencies", default ON, section 8) — with it off, rows proceed and a genuinely missing parent still fails at Fusion with a reportable error. Example: a PO header requires its supplier LOADED. Enforces "no transactional record without loaded master data".
(2) Within-object cascade — header, lines, locations, and distributions travel together in the same FBDI, so there is no Fusion check between them. Instead, if a parent row fails pre-validation in this pass, the failure cascades down (header → lines → locations → distributions) so no orphan children enter the file.

Three independent status levels — never conflate them

Status is tracked at three altitudes, each with its own column and vocabulary. A run is one submission of the pipeline (one or more objects); a work item is one unit of queued work inside a run (typically one object, or one FBDI partition of an object); a row is one source record moving through the STG and TFM tables. A run can be COMPLETED while individual rows are FAILED; a row is only ever judged by its own STATUS.

Run — RUN_STATUS · DMT_PIPELINE_RUN_TBL

StatusMeaning
QUEUEDRun submitted; waiting for the queue poller to pick it up.
IN_PROGRESSAt least one work item is not yet finished (waiting, processing, or polling).
COMPLETEDAll work items finished; every record accounted for with no failures.
COMPLETED_ERRORSAll work items finished; some rows ended FAILED (with reportable errors).
FAILEDThe run itself could not finish — an unaccounted-for record or infrastructure failure. Work items SKIPPED because of that failure don't change the status further: their records were never selected, so they are untouched, not unaccounted (decided 2026-07-07).
NO_ROWS_PROCESSEDRun finished and every work item selected zero rows — nothing in the whole run matched the scenario/mode. An individual zero-row item is simply DONE with zero counts (grey tile); it does not make the run NO_ROWS_PROCESSED (decided 2026-07-07).

Work item — WORK_STATUS · DMT_WORK_QUEUE_TBL

StatusMeaning
PENDINGCreated; waiting for its DEPENDS_ON predecessor to finish.
READYDependencies satisfied; eligible for pickup by the heartbeat.
PROCESSING (end state — code/UI still say VALIDATING, P2 §12)A child job is running the item's data phase end-to-end: pre-validate the object's STG rows (dependency checks) → transform STG → TFM (prefix, derivations) → post-validate TFM rows (content checks) → generate the FBDI/HDL file (TFM → DMT_FBDI_CSV_TBL → DMT_FBDI_ZIP_TBL) → submit the load to Fusion. One status covers the whole phase because it runs as a single child job (DMT_QUEUE_WORKER_PKG.EXECUTE_ONE); the item leaves PROCESSING when the load is submitted (→ LOADING/AWAITING_LOAD) or the phase fails. See "Pipeline stages defined" above for what each step checks.
LOADINGLoad submission in progress (ZIP → UCM upload → ESS job submit).
AWAITING_LOADESS load job submitted; heartbeat is polling for completion.
AWAITING_IMPORTImport ESS job submitted; polling.
AWAITING_POSTRUNPost-run ESS step (per-object post-processing job) submitted; polling. Today only Assets has one (per-book Prepare + Post); the per-object post-run job is a catalog column (section 6). Distinction, stated to remove ambiguity: a post-run job is an ESS job that further processes the imported data. Reporting and reconciliation steps — including Projects' post-import reporting and every BIP validation report — are not post-run jobs; they belong to the Reconcile phase.
RECONCILINGBIP reconciliation child job is running.
DONEItem finished and every row is accounted for — each row ended either LOADED (confirmed in Fusion base tables) or FAILED with a reportable error. Rows may have failed; DONE means nothing is unexplained.
FAILEDItem ended with at least one row unaccounted for (no positive success and no positive failure — e.g. reconciliation could not resolve it), or the item hit an unrecoverable infrastructure error. An ESS job ERROR does not short-circuit to FAILED — the item still reconciles, and only goes FAILED if rows remain unexplained.
SKIPPEDNot run because a dependency failed.
The accounting rule. A work item is DONE if and only if every record is accounted for (base-table LOADED or interface-errored FAILED); it is FAILED only if any record is unaccounted for. Row failures alone never fail the work item — unexplained rows do.

How users run and re-run data — run modes + prefixes

Three run modes (RUN_MODE on the run), selected at submission. Combined with prefixes, they cover the real load lifecycle:

ModeSelectsTypical use case
NEW (default)Staging rows with status NEW — never yet attempted. TRANSFORMED and FAILED rows are untouched. (RETRY and the ALL-mode reset are retired — run modes drive the selection predicates directly.)Incremental loading. Load 10% of the data as a sanity check and run it; once it proves out, load the other 90% and run again — only the new rows are picked up. Repeat as more data arrives.
FAILEDTwo sources, one union: staging rows whose status is FAILED (died before the transform table), plus staging rows whose most recent TFM attempt ended FAILED (rejected downstream — staging still says TRANSFORMED because downstream outcomes are never written back to staging). Must use the most recent attempt: a record that failed in run 57 and loaded in run 58 is not selected. NEW rows are ignored.Retry after a fix. 100% loaded but 20% failed on a Fusion setup miss; fix the setup, run FAILED mode — just the failures re-run, the 80% that loaded stay untouched.
ALL (requires a scenario)Every row in the scenario, selected directly by the run-mode parameter in the selection predicates (decided 2026-07-07) — no status reset, no extra procedure. The run still gets a new prefix. Current code instead calls reset_scenario_status to flip rows to a RETRY status — replace with mode-driven predicates and retire both the procedure and the RETRY status (P3, section 12).Full regression re-run. After fixing the setup miss you want to reload all 100% to prove the fix didn't break the rows that previously worked. The new prefix keeps the re-load from colliding with the earlier load in Fusion.

Why prefixes make this safe. Every test run stamps a fresh numeric prefix onto business keys at transform, so the same source records can be loaded into Fusion repeatedly — each run creates distinct Fusion records, and reconciliation joins each run's rows by its prefix, so runs never blur together. A production cutover runs once, without a prefix, loading the data as-is.

Glossary

2 · Execution engine (queue & heartbeat) VERIFIED 2026-07-04

How queued work actually gets executed. For what each pipeline stage does to the data, see Pipeline stages defined in the overview; for the status vocabularies, the status tables there.

The heartbeat

One permanent DBMS_SCHEDULER job, DMT_QUEUE_POLLER, fires DMT_QUEUE_PKG.HEARTBEAT_TICK every 60 seconds (C_POLL_INTERVAL). Each tick is deliberately lightweight:

  • Promotes PENDING → READY when a work item's DEPENDS_ON predecessors are DONE — or, when the run was submitted with On Failure = Continue, when they are terminal (DONE or FAILED): dependents launch anyway and per-row dependency validation sorts out individual rows (decided 2026-07-07).
  • Handles data-dependent splits — partitions that can only be known after transform (Assets per book) get their per-partition work items created mid-run, and the Plan preview marks them "split at run". Partitions knowable from staging data (PO by business unit, GL by ledger) are computed by the Plan step and created at Confirm Submit — so "creates precisely what is shown" holds (stated 2026-07-07; per DMT_CEMLI_SPLIT_CFG).
  • Checks ESS state for items in AWAITING_LOAD / AWAITING_IMPORT / AWAITING_POSTRUN.
  • Spawns one-shot child jobs for the heavy work (below).
  • Rolls the run status up from its work items — the mapping is exactly the run-status definitions in the Overview's status table: each status's stated meaning is its rollup condition — defined once, there. Precedence (stated 2026-07-07): a run is IN_PROGRESS while any work item is unfinished — failures do not settle the run status early — and the terminal statuses apply only once every item is terminal.

Concurrency rule (decided 2026-07-07): an object may be part of only one active run at a time. Enforced at submission — a run whose selection includes an object already in an active run is rejected with a clear message naming the object and the blocking run. This removes row-level race conditions between overlapping runs by construction; background-job limits are the database scheduler's concern.

Child jobs — the heavy work

Heavy work runs in one-shot DBMS_SCHEDULER child jobs in a separate package, DMT_QUEUE_WORKER_PKG (separate so the heartbeat never self-deadlocks on its own package in the library cache):

ProcedureDoes
EXECUTE_ONERuns the item's data phase: pre-validate → transform → post-validate → generate → submit load (the PROCESSING work status — currently named VALIDATING in code, P2 §12).
POLL_ONEOne ESS check; advances the work item through the AWAITING_* states. Transient SOAP faults retry next tick; EXPIRED is not treated as terminal.
RECONCILE_ONERuns BIP reconciliation and settles every row LOADED / FAILED.

The worker calls DMT_LOADER_PKG.RUN_* with g_async_mode=TRUE (return-after-submit instead of the loader's own inline poll loop).

HDL work items use a subset of the statuses (decided 2026-07-07): PROCESSING (validate → transform → generate the .dat) → LOADING (upload + submit, one REST call each) → AWAITING_LOAD (poll the dataset to a terminal state; EXPIRED retries next tick) → RECONCILING (HDL failure messages + the object's Contract v1 report) → DONE or FAILED by the accounting rule. AWAITING_IMPORT and AWAITING_POSTRUN are never used by HDL items — HDL is one combined job with no post-run step.

REST-loaded work items (Banks, ARReceipts) use the shortest subset (decided 2026-07-07): PROCESSING (validate → transform) → LOADING (the record-at-a-time API calls, each recording its outcome as it goes) → DONE or FAILED by the accounting rule. No AWAITING states at all — there is no job to await; the load is the reconciliation.

Timeouts (decided 2026-07-07)

The polling timeout is configurable — ESS_POLL_TIMEOUT_MINUTES in DMT_CONFIG_TBL (default 30), maintained at the administrator level: raise it before a 100,000-row load, keep it short for everyday runs. A timeout is a trigger for the failure path, never a verdict: when it expires, every GENERATED row of the file is marked FAILED with a [LOAD_ERROR]-tagged timeout message (the all-or-nothing rule — an unfinished load committed nothing trustworthy), and reconciliation still runs — if the job actually completed after we stopped watching, reconciliation finds the records in Fusion and marks them LOADED, overriding the precaution. The work item ends DONE or FAILED by the accounting rule alone, never directly from the timer. A row rescued this way is LOADED with historical error text — a defined, legal state: status is always current, error text is the accumulated history (the override appends a confirmation note rather than erasing the timeout message), and the UNRECONCILED derivation is unaffected (it reads FAILED + empty text).

Human guidance, not engine behavior: a job still in WAIT/RETRY past ~20 minutes almost always means a bad ParameterList — an operator (or an investigating session) should stop watching and check the parameters rather than wait out the timeout. The engine never reads this number; nothing is marked differently because of it.

A fresh build_local_db.sh --fresh (or db_full/tools/build_local_db.sh --fresh) builds the full schema clean, but the database is not yet operational: three runtime settings are deliberately left unset because they are secrets or environment-specific and must never live in committed install files. The three manual settings required after every fresh install:

  1. Global Fusion password — DMT_CONFIG_TBL.FUSION_PASSWORD ships as the placeholder ***MASKED-SET-ME***. Set it to the real password of the global FUSION_USERNAME user (currently fin_impl).
  2. Per-object credential overrides — every row in DMT_ERP_INTERFACE_OPTIONS_TBL that carries a FUSION_USERNAME (12 rows: Suppliers / PurchaseOrders / Requisitions / BlanketPOs / Contracts + the five supplier objects run as calvin.roth; Items / ItemCategories / MiscReceipts run as SCM_IMPL) also ships with FUSION_PASSWORD = ***MASKED-SET-ME***. After it runs, a live pipeline (e.g. DMT_SUBMIT_RUN_V2('P2P', ...)) can reach Fusion.

Secrets stay out of git. The passwords are never committed; the placeholder is what ships, and setup_runtime_config.py injects the real values at runtime from connections.json. 5 · Error handling VERIFIED 2026-07-04

How failures are captured, attributed, and prevented from being mistaken for success.

  • Prefix convention. ERROR_TEXT is tagged by the stage that raised it — see the tag table below for when each is written and by which package group.
  • Accumulate, never overwrite. ERROR_TEXT is a CLOB appended via DMT_UTIL_PKG.APPEND_ERROR at every step.
  • Per-row attribution. Errors attach to the specific header/line/distribution row that failed — no rolling everything up to the header with a generic "parent failed" cascade.
  • Cascade to children. A FAILED parent (e.g. PO header) cascades to its children (lines/locs/dists), each row carrying why.
  • absence ≠ LOADED. 0 BIP rows is never treated as success — it is investigated. A row that comes back with neither positive Fusion id nor an error is UNRECONCILED — a derived classification (FAILED + no ERROR_TEXT), not a stored status; see the status vocabulary in the Overview.
  • Object-status accounting. An object is DONE iff every record is accounted for (base-loaded OR interface-errored); FAILED only if any record is unaccounted. A Load ESS ERROR marks all GENERATED rows FAILED and routes to reconcile — it never short-circuits.

How each row's outcome is resolved (end state)

After the load ESS job finishes, reconciliation (DMT_{obj}_RESULTS_PKG.RECONCILE_BATCH) decides every TFM row's fate. In priority order:

  1. Positive base-table match → LOADED, no error. The BIP report finds the row in the Fusion base table (Tier 2) with its Fusion ID. Nothing is appended to ERROR_TEXT.
  2. Interface error table → FAILED + [FUSION_ERROR]. The row is still in the Fusion interface table with rejection text (Tier 1); the BIP report returns that text and it is appended per row. This is the standard path for FBDI objects (AP, PO, AR, GL, suppliers, customers, …).
  3. Purging imports → FAILED + [IMPORT_REPORT] (fallback). The Projects-family imports (PJC_TXN_XFACE_STAGE_ALL: Projects, Expenditures) purge rejected rows from the interface table, so BIP can return 0 rows even when rows failed. The reconciler then parses the ESS import report via DMT_IMPORT_REPORT_PKG.PARSE_ERRORS and matches errors back to TFM rows on the row identifier (ORIG_TRANSACTION_REFERENCE).
  4. REST-loaded objects (Banks, ARReceipts) → outcome from each API response — success with the Fusion id, or a specific error, per record, recorded directly on the TFM row; no BIP step (low-volume master data).
  5. HDL objects (HCM) → failures from the HDL response, positive success from a Contract v1 BIP report (decided 2026-07-07). The failure tier is free and per-record: every failing line comes back as a message identifying the record by SourceReference001 — the object's business key, the same value as the report contract's RECORD_KEY — and is marked FAILED with [FUSION_ERROR]. "Dataset finished and the record is not in the failure list" is only the interface-tier signal — never sufficient for LOADED on its own. LOADED requires the base-tier proof: a Contract v1 reconciliation report per HDL object returns every record's key and Fusion-assigned id from the HCM base tables in one bulk call (a per-record REST lookup was rejected: at conversion volume — tens of thousands of workers — one call per record takes hours; the partial 3-object REST lookup retires). Records that are absent from the failure list but also missing from the report fall to the [RECONCILE_ERROR] sweep.
  6. Still unaccounted → FAILED + [RECONCILE_ERROR]. The unmatched-row sweep: not found in the BIP result, base table, or import report — "cannot verify import outcome." A row left FAILED with no ERROR_TEXT surfaces as the derived UNRECONCILED classification (Overview) and fails the object's accounting rule. End state — this sweep is enforcement, not per-object code: it runs as a shared common utility invoked by DMT_QUEUE_WORKER_PKG.RECONCILE_ONE after each object's RECONCILE_BATCH, driven by the TFM-table catalog, so no reconciler can omit it. Today it exists inline in only 9 of ~42 reconcilers (P2, §12).

When a load fails, assume our code or our data is wrong — download the Fusion job's diagnostics first and find the defect on our side; "the instance is broken" is the last hypothesis, not the first.

Which columns hold what (stated to remove ambiguity): the outcome word (LOADED / FAILED) goes in the TFM row's STATUS column, and only there. The Fusion identifier goes in the object's FUSION_*_ID column(s) (e.g. FUSION_INVOICE_ID) — IDs only, never words: a FUSION_*_ID column never contains "LOADED" or any other text. Rejection text goes in ERROR_TEXT.

Independently of row status, DMT_LOADER_PKG calls DMT_IMPORT_REPORT_PKG.PARSE_AND_LOG_ERRORS after each FBDI load to write all import-report errors to DMT_LOG_TBL — run-level diagnostics only; it does not touch TFM rows.

Volume — keyset pagination (decided 2026-07-07). Every data model orders its rows by RECORD_KEY and returns at most P_CHUNK_SIZE rows after P_AFTER_KEY. The shared fetch function loops — first call with an empty cursor, each next call passing the last key received — feeding each page to the parser as it arrives, until a page returns fewer rows than the chunk size. Memory is bounded to one chunk regardless of total volume; a page-count cap derived from the run's generated-row count stops a misbehaving report from looping forever. Keyset was chosen over offset-plus-overlap paging: the cursor is exact (no gaps, no duplicates, no drift — and the reconciliation population is fixed anyway once the load and import jobs finish), and every page costs the same. A scheduled bulk file export was considered and rejected as heavier machinery (delivery infrastructure, job polling, giant-file parsing) for the same result.

Outcome semantics. A TFM row is LOADED only from a BASE / SUCCESS / FUSION_ID NOT NULL row; INTERFACE/SUCCESS is corroborating, never sufficient. Zero report rows is never success — the reconciler logs a warning, rows stay unaccounted, and the accounting rule fails the work item. A SOAP fault raises immediately, never a silent retry.

HDL objects conform too (decided 2026-07-07): each HDL object gets a Contract v1 report whose rows are the BASE tier (key + Fusion id) — its ERROR rows are optional, because HDL failures arrive per-record in the HDL response and are tagged [FUSION_ERROR] before the report runs. The report's RECORD_KEY is the same business key written into the .dat as SourceReference001 — one key definition per object serving the screens, BIP matching, and HDL matching.

One generic parser. The ~25 bespoke parse blocks collapse into one shared parser reading only the seven standard columns, driven by per-object registration added to DMT_BIP_REPORT_TBL: TFM_TABLE, FUSION_ID_COLUMN, and CONTRACT_VERSION — the key-building expression (RECON_KEY_SQL) lives on the per-object registry, shared by every consumer (decided 2026-07-07) (1 = conforms; 0 = legacy — the flag lets standard and legacy coexist during per-object migration, each step gated by the regression scenario).

Showing final outcomes next to staging data (decided 2026-07-07): screens that want a "Loaded" column on staging lists read a view joining each staging row to its most recent TFM attempt — computed at query time from the single source of truth. No process ever writes downstream outcomes back to the staging table.

Credentials (decided 2026-07-07)

How a credential is resolved — master with per-object overrides. One master Fusion account lives in DMT_CONFIG_TBL and is the default for every Fusion call. Individual objects override it where a load or a BIP report call needs different permissions — every object's override lives in one place: a credential-override column on the object registry (the one-row-per-object registration table the catalog-driven dispatch reads — decided 2026-07-07; the existing FBDI override rows in the ERP interface options migrate there, so FBDI, HDL, and REST-loaded objects all resolve identically). The resolution rule is: use the object's override if one exists, otherwise the master. One consequence is a hard rule learned in production: job-status polling must use the same account that submitted the job — Fusion will not show one user's job to another, so the poller resolves credentials exactly the way the submitter did.

  • Inventory: every credential the tool holds (master account, per-object overrides, wallet) is listed with its storage location — no credential exists that isn't in the inventory.
  • Display: the Administration screen shows credentials masked, with an explicit reveal action — never in plain lists or exports.
  • Rotation: one procedure updates every place a credential is cached, in one operation (this exists operationally today); partial rotation is how loads break with 401 errors.
  • Never in git: database exports, seed files, and repository content never contain credential values — the deploy snapshot masks them, and a committed credential is a defect (one was found and is tracked as issue #36).

The top-level runner signature is identical for every object; only the object literal changes. Mandatory: p_run_id. Optional (with defaults): scenario, run-mode, and a lookup-refresh skip used when preflight already refreshed:

The body is three lines every time — resolve the scenario name to an id, then call the shared engine:

PROCEDURE RUN_SUPPLIERS (p_run_id, p_scenario_name DEFAULT NULL, p_run_mode DEFAULT 'NEW', p_skip_bu_refresh DEFAULT FALSE) IS l_dummy BOOLEAN; v_scenario_id NUMBER; BEGIN resolve_scenario(p_scenario_name, v_scenario_id); l_dummy := run_one_object_type(p_run_id, 'Suppliers', v_scenario_id, p_run_mode, p_skip_bu_refresh); END;

The upstream-validation parameter (owner requirement, 2026-07-14) is threaded run → queue → dispatch → the process-one body as p_validate_upstream (Y/N) and is never hardcoded inside a validator. Objects migrating their own dependencies in the same run (PO → Suppliers) default it ON; objects whose references pre-exist in Fusion (APInvoices → supplier, ARInvoices → customer) default it OFF. This single parameter retires the per-validator hardcoded dependency check.

The five ordered steps (the process-one body)

  1. Refresh lookups & ERP options. If NOT p_skip_bu_refresh: DMT_UTIL_PKG.REFRESH_LOOKUPS. Then get_erp_options(p_cemli_code, x_ucm_account, x_job_name, x_interface_details_id) fills the job/account/interface locals.
  2. Validate. Call the object's validator. The upstream-dependency arm runs only if p_validate_upstream = 'Y': join each TFM row's reference to the upstream object's TFM, require TFM_STATUS='LOADED', mark unmatched rows FAILED with [PRE_VALIDATION]. When N, skip it entirely (the reference already exists in Fusion). COMMIT.
  3. Transform STG → TFM. Select STG rows by the run-mode status set (NEW → NEW,RETRY; FAILED → FAILED; ALL → whole scenario) and scenario id; write each to TFM as STAGED, deriving but never overwriting; flip the STG row to TRANSFORMED. COMMIT. Data is never re-inserted per run — it is loaded once as a scenario and re-run in place under a new prefix.
  4. Generate + submit load. Call GENERATE_FBDI (per partition value if partitioned — see below). Empty zip ⇒ log and stop cleanly. Else GET_CEMLI_CREDENTIALS, then l_load_ess_id := SUBMIT_LOAD(...), free the zip, stamp PARAMETER_LIST on DMT_FBDI_ZIP_TBL. In ASYNC mode return here with g_load_ess_id set — the worker sets WORK_STATUS='AWAITING_LOAD' and polling/reconcile run later in POLL_ONE/RECONCILE_ONE. In SYNC mode continue inline: poll load, poll import, reconcile.
  5. Reconcile — positive confirmation (Rule #1). RECONCILE_BATCH runs the BIP report, parses it, and flips GENERATED → LOADED for rows confirmed in the base table with a real Fusion id; unconfirmed/rejected → FAILED. Then the accounting gate (apply_accounting_gate) — the only writer of WORK_STATUS='DONE' — counts every TFM row: DONE only if total − loaded − failed_with_error = 0, else FAILED.

The one variation — partitioning

If a value comes from the uploaded file and is a positional ESS ParameterList argument, it is a partition key: emit one FBDI + one load/import per distinct value so the CSV column and the parameter always agree. The shape: the generator exposes the key as a trailing param (p_ledger_name, p_batch_id, p_bu_name…), default NULL = whole run, which filters every CSV query and the STAGED→GENERATED update. The loader opens a cursor over SELECT DISTINCT <key> FROM DMT_<STEM>_TFM_TBL WHERE RUN_ID=p_run_id AND TFM_STATUS='STAGED' and calls the shared nested submit_and_reconcile_one once per value; on a per-partition load failure only that partition's GENERATED rows are marked FAILED ([LOAD_ERROR]), then CONTINUE. Cursor names by family: led_rec (GL/ledger), grp_rec (AR/Customers/Req/Items/batch), bu_rec (PO), ou_rec (AP). Worked examples: GLBalances partitions DMT_GL_INTERFACE_TFM_TBL by LEDGER_NAME, resolving the id via GET_LOOKUP('LEDGER_NAME_TO_LEDGER_ID', …); Customers partitions seven DMT_HZ_*_TFM_TBL tables by a shared BATCH_ID (which itself becomes ESS arg 1), enforcing one source-system per batch. The simple path (Suppliers) is set-based with no cursor.

Status ladder & exception handling — identical for every object

Status columnLives onValues & transitions
STG_STATUSthe STG tableNEW/RETRY → TRANSFORMED (or TRANSFORM_FAILED/FAILED)
TFM_STATUSthe TFM tableSTAGED → GENERATED → LOADED (confirmed in base table) or FAILED (has ERROR_TEXT)
WORK_STATUSDMT_WORK_QUEUE_TBLREADY → VALIDATING → AWAITING_LOAD → AWAITING_IMPORT → RECONCILING → DONE/FAILED
  • Per step: EXCEPTION WHEN OTHERS THEN DMT_UTIL_PKG.LOG_ERROR(...); RAISE; — log then re-raise; the worker's WHEN OTHERS flips WORK_STATUS='FAILED'.
  • ERROR_TEXT is append-only via APPEND_ERROR. Tags: [PRE_VALIDATION], [LOAD_ERROR], [FUSION_ERROR], [RECONCILE_ERROR]. Never reset on reprocess.
  • Marking FAILED is always scoped to RUN_ID + TFM_STATUS='GENERATED' (+ partition column when grouped); reconcile is guarded AND TFM_STATUS NOT IN ('FAILED','LOADED') so a proven-LOADED row is never flipped back.
  • A job-level ESS ERROR routes to reconcile, never short-circuits: mark all GENERATED rows FAILED with [LOAD_ERROR], and still set RECONCILING so BIP can flip genuinely-confirmed rows back to LOADED.
  • The accounting gate is the single terminal decision. Nothing is DONE until every TFM row is accounted (loaded, or failed with non-empty ERROR_TEXT). A FAILED row with empty ERROR_TEXT counts as unaccounted → the whole item is FAILED.

What a tile counts, and partitions (decided 2026-07-06)

Tile numbers count parent records only — for PurchaseOrders, orders (headers), not lines. But a parent is not successful unless all of its children loaded: an order with a failed line counts as failed, so child failures always surface in the tile numbers and color. The per-record-type split lives on the Object Detail funnel.

The rollup formula — worst child wins (decided 2026-07-07): a parent record's displayed state is the worst state across itself and all of its children, ranked unaccounted > failed > loaded. A fully clean order counts as loaded; an order with any failed line (all accounted) counts as failed; an order with any unexplained line counts as unaccounted. The three counts always sum to the parent total, and the tile palette applies to these rolled-up numbers unchanged.

Partitioned objects render one tile per partition (PurchaseOrders per Procurement BU, GL Balances per ledger) — each with its own counts and color. This is 1:1 with work items, which are already created per partition, so the tile palette stays honest per work item; the drill passes the partition key alongside the object code.

Q1 — What do a Run Detail tile's numbers count? DECIDED 2026-07-06

Decision: parent records only — but a parent is not successful unless all of its children loaded. Written into section 9.

The Run Detail screen shows one tile per object, e.g. "PurchaseOrders — 104 loaded / 15 failed". But one PurchaseOrders FBDI contains four record types with their own counts (headers 28 loaded, lines 104, locations 110, distributions 127). The document never says whether the tile shows the sum of all record types (369 loaded), one record type, or something else. The tile color rules (section 9) depend on these numbers, so the palette cannot be implemented until this is decided.

Options: (a) sum every record type — "369 loaded / 15 failed / 1 unaccounted" — honest but big numbers; (b) show only the parent record type (headers: "28 / 1") — small numbers but hides child failures; (c) sum for the color decision but display "records: 369 ✓ · 15 ✗ · 1 ?" with the per-type split on the Object Detail funnel. Suggestion: (c) — color must consider every record (a child failure must be able to turn the tile light green/red), and the funnel is where the split already lives.

Q2 — How do partitioned objects appear on screen? DECIDED 2026-07-06

Decision: one tile per partition, each with its own counts — 1:1 with work items, which are already created per partition (so each partition has its own work-queue id today; no new mechanism needed). Written into section 9.

A grouped object splits into one work item per partition — PurchaseOrders per Procurement BU, GL Balances per ledger (config table DMT_CEMLI_SPLIT_CFG). Each partition gets its own ZIP and its own ESS jobs. But the Object Detail screen receives only a run id and an object code — no partition. So: one tile per object or one per partition? Whose ESS job chips does Object Detail show? Does the funnel sum across partitions?

Options: (a) one tile per partition ("PurchaseOrders — US1", "PurchaseOrders — US2") — matches the work-item granularity exactly, so the tile palette is honest per work item, and Object Detail receives the partition key and shows that partition's jobs and funnel; (b) one tile per object, aggregated, with a partition list inside Object Detail. Suggestion: (a) — the tile represents a work item everywhere else in the design; keeping that 1:1 avoids inventing an aggregation layer. Requires passing PARTITION_KEY alongside the object code in the drill.

Q3 — FAILED-mode re-runs clear ERROR_TEXT DIRECTION DECIDED 2026-07-06

Decision (updated 2026-07-07): the stage→transform result table records failures only — success is the TFM row itself (run-stamped, timestamped) — and the STG row carries a forward-only status (NEW → TRANSFORMED or FAILED, reflecting the most recent stage→transform attempt, written only by that step — final decision 2026-07-07). Messages never go on staging; downstream outcomes are never written back to staging; FAILED-mode selection combines staging FAILED rows with rows whose most recent TFM attempt failed; dependency checks read the parent's TFM row. Nothing to clear, full history per run. Written into section 5. Downstream impacts to accept: (1) STG ERROR_TEXT retires — any screen showing a staging row's error must join the outcome table for the latest attempt instead; (2) volume — one row per failed record per attempt; covered by the retention policy (section 6), and STG/TFM stay lean; (3) downstream outcomes are never written back to staging — staging status is written only by the stage→transform step; (4) the name stands as-is.

The Overview run-mode table says FAILED mode re-attempts failed rows and "ERROR_TEXT is cleared". Error handling (section 5) says ERROR_TEXT "accumulates, never overwritten". The status vocabulary calls the ALL-mode reset "the one sanctioned status reset". The code really does clear (supplier transform):

IF p_reprocess_errors THEN
  UPDATE DMT_POZ_SUPPLIERS_STG_TBL
  SET    ERROR_TEXT = NULL, LAST_UPDATED_DATE = SYSDATE
  WHERE  STATUS IN ('FAILED', 'TRANSFORM_FAILED');
END IF;

Options: (a) declare FAILED-mode clearing the second sanctioned reset — the STG row's ERROR_TEXT describes the current attempt only, and history is preserved anyway: every attempt's errors are in the activity log, and (once built) the run-stamped error table keeps pre-transform failures per run; (b) never clear — append an attempt separator instead ("—— retry, run 117 ——"), so one column holds the full history at the cost of ever-growing CLOBs and confusing "is this error current?" reads. Suggestion: (a), with the Overview and section 5 reworded to say exactly that: current-attempt errors on the row; full history in the log and the run-stamped error table.

Q6 — What is the life of an HDL work item? DECIDED 2026-07-07

Decision: HDL items use the status subset PROCESSING → LOADING → AWAITING_LOAD → RECONCILING → DONE/FAILED (no import or post-run states — written into section 2). Failures are positive and per-record from the HDL messages; positive success requires the object's Contract v1 BIP report (bulk key + Fusion id from the HCM base tables) — the not-in-the-failure-list complement rule is never sufficient alone, and the per-record REST lookup was rejected on volume (hours at conversion scale). Written into section 5; build work in the backlog.

The work-item statuses are shaped around ESS jobs and BIP reconciliation (AWAITING_LOAD / AWAITING_IMPORT / RECONCILING). HDL objects load over REST and, per section 1, "HDL reconciliation isn't built yet". So which statuses does an HDL item pass through, what does DONE mean for the 14 HDL objects, and under the accounting rule (every record positively accounted for) can one ever legitimately reach DONE today?

Suggestion: define the HDL lifecycle as VALIDATING → LOADING (REST upload + submit) → AWAITING_LOAD (poll the dataset status) → RECONCILING (retrieve per-row results) → DONE/FAILED — the reconcile step already has a concrete mechanism to specify against, the HCM REST messages endpoint the utility package documents:

-- dmt_hdl_util_pkg.pks
--  Errors: GET /hcmRestApi/resources/11.13.18.05/dataLoadDataSets/{RequestId}/child/messages

And state plainly: until that reconcile step is built, an HDL work item may not be marked DONE — it ends FAILED with its rows unaccounted, which is the truthful outcome. That closes the loophole where an HDL "load succeeded" could be mistaken for records confirmed.

Q11 — The cross-run dependency check and the dependent prefix don't compose into one algorithm

The situation, step by step. Suppliers were loaded in run 42, which used prefix 1042 — so the supplier the source system calls "ACME Corp" exists in Fusion as "1042ACME Corp". Weeks later you run PurchaseOrders as run 57 (its own prefix, 1057). Each PO's staging row references its supplier by the raw name, "ACME Corp". Two things must now both work:

  1. The dependency check — "is this PO's supplier loaded?" — must compare the values exactly as they appear in the source data — no prefix applied: the PO's "ACME Corp" against the supplier staging row's "ACME Corp" with STATUS='LOADED'. Comparing the raw name to the prefixed value ("1042ACME Corp") can never match — that was exactly the ProjectBudgets defect fixed 2026-07-03.
  2. The reference written into the FBDI file must carry the supplier run's prefix, not the PO run's: the PO file must say supplier "1042ACME Corp" (what Fusion actually contains), not "ACME Corp" (doesn't exist in Fusion) and not "1057ACME Corp" (wrong run's prefix — also doesn't exist).
-- 1. VALIDATE (source values as-is, no prefix, against the parent's STAGING row):
SELECT 1 FROM dmt_poz_suppliers_stg_tbl s
JOIN  dmt_poz_suppliers_tfm_tbl t ON t.stg_sequence_id = s.stg_sequence_id
WHERE s.supplier_name = po.supplier_name  -- 'ACME Corp' = 'ACME Corp' (source values, no prefix)
AND   t.status = 'LOADED';  -- loaded-ness is proven on the TFM row, never on staging

-- 2. TRANSFORM (write the PARENT run's prefix into the reference):
tfm.vendor_name := p_dependent_prefix || stg.supplier_name;  -- '1042' || 'ACME Corp'

-- 3. p_dependent_prefix ('1042') is picked at submission from the prefix-history view:
--   "which run loaded the suppliers I depend on, and what prefix did it use?"

DECIDED 2026-07-07: the pattern is the standard, generalized — the check accepts a LOADED parent from any run, the reference prefix comes from the parent's most recent LOADED row (the dependent-run picker is an optional override), and dependency validation can be turned off per run (submission screen, section 8).

Q12 — Retention DECIDED 2026-07-06

Every run stores full CSV content (CLOBs), full ZIPs (BLOBs), ESS output files, and an ever-growing activity log that survives rollbacks by design. No retention or purge policy exists anywhere.
