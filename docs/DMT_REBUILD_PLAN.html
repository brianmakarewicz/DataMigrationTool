1. Feedback & Recommendation

Bottom line: the approach is sound, and you are closer to it than you think — but reframe it. This is not a restart; it is a re-platform plus targeted refactor. The feature/db-full-install work already produced the two hardest ingredients: a proven greenfield install (1,055 objects, 100% parity between ATP and local Docker per COMPARE_REPORT.md) and a git-first deployment discipline. The new repository should be seeded from db_full/, not from the old repo's history. That single decision converts "restart" from a 6-month rewrite risk into a 4–6 week migration.

What I agree with

  • New repository with a curated seed. The current repo carries drift baggage: 73 invalid *_IMP_RPT_V views, a legacy 40 KB deploy.sql, 9 stale APEX export snapshots (apps 119–155), dead runner packages and abandoned adaptor stubs already marked RETIRE in the design doc. Starting a clean repo lets you apply the design doc's open-items naming sweep at port time instead of as churny in-place renames, and gives you clean git history for the approval agent to review.
  • New ATP instance. Full isolation from the frozen instance is the right call given the drift history. It also forces the install script to prove itself on a truly empty instance — the exact discipline that prevents drift from recurring.
  • Docker-based local testing before ATP deploys. You already have both halves: the APEXResourceTracker two-service compose stack (Oracle Free + ORDS + APEX) and the DMT build_local_db.sh (port 1522). Merging them gives PR-level CI that compiles the whole schema on every pull request.
  • Independent approval agent on a second GitHub account. Correct separation of duties: author account opens PRs, reviewer account (which already exists) holds the required review under branch protection. You already run a cloud PR-review routine for RegressionTestScripts — the same pattern extends here.
  • 80% DDL reuse. The survey supports this. The 89 STG/TFM table pairs, control/config/queue/run tables, sequences, and most views carry over nearly verbatim; the 20% that changes is names (view _V suffix convention), the new DMT_STG_TFM_ERROR_TBL, dropped dead views, and the catalog-dispatch registry tables.
  • EBS Conversion as Phase 2. Sensible sequencing. EBSAdaptors is production-ready (61 views, 1.1M rows extracted E2E) and only needs repointing at the new ATP; ConfigurationMigrationTool is ~90% through its own Phase 1 but blocked on EBS VM networking — putting it in your Phase 2 gives time to unblock it without holding up the core rebuild.

Where I'd push back or add caution

  • Do not redesign the pipeline. The biggest risk in any "restart" is the second-system effect. The design doc's open-items backlog (its section 12 — catalog-driven dispatch, shared RUN_BIP_REPORT, error-tag conformance, funnel view, config objects into the queue) is a complete, already-agreed refactor scope. Treat that list as the entire refactor budget; anything not on it ships as-is and gets improved later.
  • Freeze the requirements before Phase 2 of the build, not before Phase 1. You noted the requirements are in flux. Infrastructure (repo, ATP, Docker, CI/CD, approval agent) is requirements-independent — start it immediately. Gate only the schema/engine port on a tagged DMT_DESIGN.html v1.0. Ten of twelve open design questions were resolved 2026-07-06/07; the two parked ones (FBL file delivery, authorization/roles) do not block the port.
  • "Freeze the current instance" only freezes half the environment. The Fusion demo pod is shared between old and new stacks. BIP reports under /Custom/DMT/, ESS credentials, and demo password rotations affect both. Decide a Fusion-side convention early (e.g., new reports under /Custom/DMT2/ or reuse in place) — see risk R4.
  • An approval agent must review, not rubber-stamp. Wire the reviewer account to run the actual gates (schema compile in Docker, regression subset, code review against the 14 coding standards) and to block on failure. An auto-approve bot that always approves is worse than no branch protection, because it launders unreviewed changes.
  • Port the regression harness in Phase 1, not at the end. "Testing on each object, not exhaustive" only stays cheap if the existing suite (dmt_regression_run.py, seeded 1–2 good + 1 bad rows per CEMLI, the dmt-regression-tester agent) moves over early so every wave of object migration is verified the day it lands — per Rule #1, LOADED in Fusion base tables and FAILED with reportable errors, never interface-only.
  • Budget honestly for the objects that were never finished. Of 45 CEMLIs, 15 are E2E LOADED, 5 are wired but unproven, 17 are DDL-only (mostly FBL config objects with the parked file-delivery question), 3 are blocked by the demo instance, and the rest are HDL not-built or dormant. "80% reusable" is true of DDL; it is not true of finished pipelines. The roadmap below reflects that.
Recommendation: Proceed, with the reframing above. Sequence: infrastructure first (requirements-independent), then freeze requirements v1.0, then port engine + objects in waves with per-wave regression, then EBS Phase 2. If you decide not to proceed, the fallback is nearly the same work on a branch: the db_full worktree + the same open-items backlog executed in place — you'd save ~3–4 weeks of infrastructure but keep the drifted repo history and shared instance. The isolation benefits justify the new-stack approach if you're working on this full time.

4. Build Order of Operations & Incremental Test Plan

This is the construction sequence inside Phases 2–3: what gets created in which order, what can be built in parallel, and — critically — how each layer is proven before the next layer goes on top. Nothing waits for a big-bang test at the end: every stage has its own verification, and the full regression scenario is the gate before any APEX work begins. The ordering follows the dependency order already encoded in db_full/install.sql (sequences → tables → foreign keys/synonyms → views → package specs → procedures → package bodies → scheduler jobs → grants → seed data).

Two guiding rules for this section. (1) Common utilities first: every per-object package calls DMT_UTIL_PKG for logging/errors and the shared ESS/BIP/REST plumbing, so those are built and unit-tested before any CEMLI code exists. (2) Docker before Fusion: everything that can be proven without Fusion credentials (DDL, CSV parsing, FBDI generation, queue mechanics) is tested in the local Docker instance via CI; only the Fusion-facing calls (ESS, UCM, BIP, REST) are tested against the live demo pod, and each of those gets its own isolated unit test before any pipeline uses it.
